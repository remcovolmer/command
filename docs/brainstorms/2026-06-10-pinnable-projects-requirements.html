Pinnable projects vervangen het workspace-type

Datum: 2026-06-10 Branch: feat/pin-project Scope: Deep — feature Status: Klaar voor planning

Samenvatting

Pinnen wordt een vlag op elk project; het workspace-type — dat positie en feature-set onterecht koppelde — verdwijnt.

Vandaag is workspace één van drie project-types, dat tegelijk twee losse dingen regelt: bovenaan-vastgezet staan én een uitgeklede feature-set. Die koppeling klopt niet. We splitsen ze: pinnen wordt een orthogonale eigenschap die elk project naar een eigen sectie bovenaan tilt, terwijl het type (project of code) blijft bepalen welke tools het project heeft. Een gepind code-project houdt dus z'n worktrees en git-tab — iets wat vandaag onmogelijk is.

Probleem

Het workspace-type dwingt een valse keuze af: wil je iets bovenaan, dan lever je dev-tools in.

Situatie

Command kent drie project-types. workspace staat altijd bovenaan in een eigen sectie, met ster en altijd-zichtbare chats.

Complicatie

"Bovenaan staan" zit vastgeklonken aan het type. Een code-project met worktrees en git kun je niet bovenaan pinnen zonder die tools te verliezen. Het type doet twee dingen tegelijk.

Resolutie

Maak pinnen een losse vlag. Positie en feature-set worden onafhankelijk: elk project — project óf code — is pinbaar en behoudt z'n eigen gedrag.

Wat we bouwen

Twee assen vervangen drie types: een type-as (feature-set) en een pin-as (positie).

Nu — type bepaalt alles

type: 'workspace' | 'project' | 'code'

★ workspace  → bovenaan + uitgekleed
  project    → in lijst + files/Claude
  code       → in lijst + full dev tools

positie & tools zijn gekoppeld

Straks — pin staat los van type

type:   'project' | 'code'
pinned: true | false

PINNED
  ★ 💻 command     code: worktrees + git
  ★ 📁 klant-docs  project: files + Claude
PROJECTS
  💻 andere-repo
  📁 losse-notes

De feature-set per type verandert niet. Wat verandert is dat de pin-eigenschap er los bovenop komt en dat elk type pinbaar wordt:

Het datamodel-detail (exacte veldnaam, hergebruik van de bestaande sortOrder voor volgorde binnen de pinned-sectie) is een planning-beslissing. Conceptueel: één pin-vlag op Project, volgorde via bestaande sorteer-infra.

Functionele requirements

Pinnen is overal beschikbaar, persistent, en laat het type-gedrag volledig intact.

  1. Pin is een eigenschap van elk project, onafhankelijk van het type. Zowel project- als code-projecten kunnen gepind worden. Een gepind code-project behoudt worktrees, git-tab, sidecar en GitHub-menu volledig.
  2. Een eigen "Pinned"-sectie bovenaan de sidebar toont alle gepinde projecten. De sectie is alleen zichtbaar bij ≥1 pin (zoals de huidige Workspaces-sectie).
  3. Gepinde projecten renderen via de bestaande SortableProjectItem — niet via een apart codepad. Daardoor erven ze automatisch alle type-specifieke UI (worktree-knop, git, indicatoren).
  4. Drie manieren om te (de)pinnen: een ster-toggle bij hover op de project-rij, een rechtsklik-actie "Pin to top" / "Unpin", en een hotkey Ctrl+Shift+P voor het actieve project. Hotkey toevoegen aan src/utils/hotkeys.ts + handler in src/App.tsx + documenteren in CLAUDE.md, conform de hotkey-vereiste.
  5. Volgorde binnen de Pinned-sectie is herordenbaar via drag-and-drop, consistent met de bestaande sorteerbare projectenlijst.
  6. De pin-status persisteert over herstart van de app (opgeslagen in userData/projects.json via ProjectPersistence).
  7. Eenmalige migratie: bij het laden converteert elk bestaand type:'workspace'-project naar type:'project' + gepind, zonder verlies van naam, pad of settings.

Bewust geaccepteerde gevolgen

Twee gedragsveranderingen zijn bewust gekozen — beide bevestigd.

1 · Always-expanded terminals vervalt. Workspaces tonen hun chats nu altijd uitgeklapt onder het project. Gepinde projecten gedragen zich straks als normale projecten: chats zijn zichtbaar wanneer het project actief is. Dit volgt rechtstreeks uit "pin = puur positie + marker".

2 · De special-case workspace-rendering wordt verwijderd. Het aparte, vereenvoudigde render-blok in src/components/Sidebar/Sidebar.tsx (~80 regels) verdwijnt; gepinde projecten gebruiken het reguliere SortableProjectItem-pad. Netto een vereenvoudiging, geen behoud.

Migratie

Bestaande workspaces worden gepinde projecten — intentie-match boven capability-match.

workspace 'klant-docs'
   → type:   'project'      (files + Claude)
   → pinned: true
   behoudt:  file explorer, Claude chats, naam, pad, settings
   verliest: git-tab + sidecar  (die workspaces nu toevallig hadden)

Gekozen boven → code omdat de bedoeling van een workspace "pinned overview" was, niet "full dev tools". Heeft een gemigreerd project later tóch git/worktrees nodig, dan is het type achteraf aan te passen — dat valt buiten deze scope.
