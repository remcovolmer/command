Tweede paneel & herontwerp van de rechterkant

date 2026-06-26 topic tweede-paneel-redesign scope Deep — feature app Command Center (Electron + React + TS)

Summary

De rechterkant wordt heringericht zodat je per gesprek je chat, je bestanden en een browser tegelijk in beeld hebt. Een nieuw tweede paneel houdt geopende bestanden en een iframe-browser vast; een dunne icoon-rail uiterst rechts opent file explorer / git / tasks / automations als auto-sluitende flyouts; een shell-drawer onderaan klap je aan/uit. Het hele werkgebied — tweede-paneel-tabs én shell — is per-chat gescoped. De split-chat verdwijnt.

Problem Frame

De huidige rechterkant dwingt schakelen waar je naast-elkaar-zien wilt, en koppelt niets aan het gesprek waar je in werkt.

Situatie

Het centrum mengt chats en editor-tabs onder één actieve tab; de file explorer is een rechterkolom; chats kun je in twee splitsen.

Complicatie

Je wilt chat + het relevante bestand + de app/HTML in beeld — per gesprek. De split-chat werkt niet lekker, en geopende pagina's hangen nergens aan vast, dus ze "reizen" mee tussen chats.

Resolutie

Een tweede paneel + icoon-rail + bottom-shell, alles per-chat gescoped. Split-chat eruit; altijd één chat zichtbaar.

Layout

"Activity-bar rechts" (variant A): chat links, tweede paneel ernaast, icoon-rail uiterst rechts, shell-drawer onderaan.

Nu

  • Centrum mengt chats + editor-tabs (één activeCenterTabId)
  • File explorer = vaste rechterkolom (4 tabs)
  • Sidecar-shells als apart rechter-paneel
  • Chat splitsbaar in 2 (werkt matig)

Straks

  • Chat-kolom (alleen chats) + tweede paneel (alleen bestanden + browser)
  • File explorer = icoon-rail met auto-sluitende flyouts
  • Shell = bottom-drawer, toggle, over chat + tweede paneel
  • Geen split; alles per-chat gescoped

Key Decisions

KD1Layout = "activity-bar rechts" (variant A)

Icoon-rail uiterst rechts met auto-sluitende flyouts (panel-iconen bovenaan, shell-toggle aan de voet), tweede paneel tussen chat en rail, shell als bottom-drawer. Gekozen boven een vaste explorer-kolom omdat het maximale ruimte geeft aan chat + content en aansluit bij de Codex-werkwijze.

KD2Per-chat werkcontext

Het hele rechts-+-onder werkgebied — tweede-paneel-tabs én shell — is gescoped op de actieve chat. De chat is de eenheid van werk: pagina's horen bij het gesprek. De rail-panelen blijven juist worktree-gescoped, want het bestandssysteem hoort bij de worktree, niet bij het gesprek.

KD3Browser = iframe, geen Chromium-webview

Lichtgewicht en bouwt voort op de bestaande HtmlPreview. Pragmatische start: lokale HTML + je eigen localhost-app werken. Externe SaaS (Gmail/Outlook) is uitgesloten omdat iframes daar geblokkeerd worden; de layout wordt zo opgezet dat een echte webview later kan instappen.

KD4Split-chat wordt verwijderd, niet herwerkt

De split werkt niet lekker en de behoefte ("chat + relevant scherm naast elkaar") wordt nu vervuld door het per-chat tweede paneel met altijd één zichtbare chat.

Requirements

Layout & navigatie

IDRequirement
R1De rechterkant bestaat uit vier zones: chat-kolom (links), tweede paneel, icoon-rail uiterst rechts, shell-drawer onderaan.
R2De icoon-rail bevat bovenaan file explorer, git, tasks en automations als iconen; een klik opent dat paneel als flyout-overlay over het tweede paneel, die weer sluit bij blur (geen vastpinnen in v1). Onderaan de rail zit de toggle voor de shell-drawer.
R3De rail-panelen zijn worktree-gescoped: ze tonen de worktree van de actieve chat. Twee chats in dezelfde worktree delen dezelfde tree.

Tweede paneel

IDRequirement
R4Een tweede paneel houdt geopende bestanden (editors/diffs) en een browser vast als peer-tabs in één tabrij.
R5Een bestand openen opent het als tab in het tweede paneel — nooit vermengd met de chat-kolom.
R6De tabs van het tweede paneel zijn per-chat: elke chat onthoudt z'n eigen set open content-tabs. Van chat wisselen wisselt de tabs mee.
R7Heeft een chat geen open content, dan klapt het tweede paneel in en vult de chat-kolom de breedte.

Browser

IDRequirement
R8De browser-tab laadt lokale HTML-bestanden én de eigen localhost-dev-app via een URL-balk, op iframe-basis.

Shell

IDRequirement
R9Een shell opent als bottom-drawer over chat + tweede paneel (niet onder de projecten-sidebar). De drawer wordt aan/uit getoggled via een knop onderaan de icoon-rail (zie R2).
R10Shells zijn per-chat: elke chat onthoudt z'n eigen shell(s); van chat wisselen wisselt de shell-drawer mee.
R11Een shell die door een chat-wissel uit beeld raakt blijft op de achtergrond doordraaien (output gebufferd) en herstelt bij terugkeer; hij wordt niet gekild.

Verwijdering

IDRequirement
R12De split-chat wordt volledig verwijderd (state, acties, pool-bescherming, UI, hotkeys). Per chat-kolom is altijd één chat zichtbaar; chats wisselen via tabs/hotkeys.

Key Flows

F1 — Bestand openen.
  1. Open een bestand vanuit de files-flyout of editor.
  2. Het verschijnt als tab in het tweede paneel van de actieve chat.
F2 — Van chat wisselen (de kern).
  1. Klik een andere chat in de chat-kolom.
  2. Het tweede paneel en de shell-drawer wisselen naar de content + shells van díe chat.
  3. De rail-panelen volgen de worktree van de chat (gelijk als zelfde worktree).
F3 — Browser gebruiken.
  1. Open de browser-tab in het tweede paneel.
  2. Typ een localhost-URL of open een lokaal HTML-bestand.
  3. De pagina rendert in de iframe.
F4 — Rail-paneel openen.
  1. Klik een rail-icoon (files/git/tasks/automations).
  2. Het paneel opent als flyout over het tweede paneel.
  3. Klik je elders, dan sluit de flyout weer.

Acceptance Examples

AE1 · dekt R6
Chat A heeft App.tsx + een browser-tab open; chat B heeft store.ts open. Wissel naar B → tweede paneel toont store.ts. Wissel terug naar A → App.tsx + browser staan er weer.
AE2 · dekt R7
Een chat zonder open bestanden of browser → het tweede paneel is ingeklapt en de chat-kolom vult de volle breedte.
AE3 · dekt R10, R11
Chat A draait npm run dev in z'n shell. Wissel naar B → de drawer toont B's shell (of niets); A's npm run dev blijft draaien. Wissel terug → A's shell verschijnt met de gebufferde output.
AE4 · dekt R2, R3
Klik het files-icoon → de tree-flyout opent over het tweede paneel. Klik in de editor → de flyout sluit. Twee chats in dezelfde worktree tonen dezelfde tree.
AE5 · dekt R8
Browser-tab: typ localhost:5173 → de draaiende app rendert. Open een lokaal .html → het bestand rendert. Typ gmail.com → blokkeert/blijft leeg (buiten scope).

Scope Boundaries

Uitgesteld (later mogelijk)

  • Echte Chromium-webview (WebContentsView / <webview>) — nodig voor externe apps, devtools en history.
  • Externe SaaS in de browser (Gmail/Outlook).
  • Browser-extra's: devtools, navigatie-history, bookmarks, meerdere browser-tabs.
  • Vastpinnen van rail-flyouts als vaste kolom.

Buiten deze wijziging

  • De linker projecten-sidebar (blijft ongewijzigd).
  • Editor-functionaliteit zelf (talen, formatting, diff-gedrag).
  • Het hook/state-detectie-systeem voor chats.

Dependencies / Assumptions

De huidige staat is geverifieerd tegen de code; deze wijziging is architecturaal, dus de relevante feiten staan hier expliciet.

  • State-splitsing. Vandaag delen chats en editor-tabs één activeCenterTabId (src/stores/projectStore.ts). Aanname: dit splitst in een actieve-chat-notie (chat-kolom) plus een per-chat actieve content-tab (tweede paneel).
  • Per-chat scoping. Content-tabs en shells worden gekoppeld aan de chat-(terminal-)id i.p.v. globaal/per-project.
  • Split-removal footprint. Te verwijderen: TerminalLayout + SplitDirection (src/types/index.ts), layouts + addToSplit/removeFromSplit/setSplitSizes (projectStore), split-bescherming in terminalPool.ts, split-UI in TerminalArea.tsx / TerminalTabBar.tsx / TerminalViewport.tsx, en de hotkeys in App.tsx.
  • Browser-iframe. De bestaande HtmlPreview (HtmlPreview.tsx) gebruikt srcdoc + sandbox zonder allow-same-origin en is daardoor niet 1-op-1 herbruikbaar voor URL-navigatie; de browser-tab vraagt een nieuwe src-gebaseerde iframe met URL-balk. webviewTag staat niet aan in webPreferences (electron/main/index.ts) — een latere webview vereist dat aanzetten of WebContentsView.
  • Shell vervangt sidecar. De bottom-shell vervangt de huidige sidecar-shells; bestaande sidecar-state migreert naar het per-chat drawer-model. Achtergrond-buffering kan leunen op het bestaande PTY-buffer-mechanisme van de terminal-pool.
  • Hotkeys. Nieuwe toggles nodig (tweede paneel, shell-drawer, rail-panelen/focus). De vrijgekomen split-hotkeys (Ctrl+\, Ctrl+Shift+\) kunnen hergebruikt worden. Conform projectconventie krijgen nieuwe features hotkeys.

Outstanding Questions

Op te lossen tijdens planning

  • Exacte hotkey-toewijzingen (toggles + focus-cyclus over chat ↔ tweede paneel ↔ rail ↔ shell).
  • Persisteert de breedte van het tweede paneel per chat of globaal?
  • Eén browser-tab per chat, of meerdere als peers? (v1 mag minimaal blijven.)
  • Migratiepad voor bestaande gepersisteerde layouts- en sidecar-state.
  • Is de rail-flyout in breedte aanpasbaar?
