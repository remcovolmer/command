Problem Frame

.html opent vandaag in Monaco als ruwe tekst — wie wil zien hoe de pagina er uitziet, moet naar de browser. De Markdown-flow bewijst het patroon al: EditorContainer routeert .md naar een Code/Eye toggle tussen Monaco en Milkdown. Voor prototypes, demopagina's en Claude-gegenereerde HTML haalt dezelfde toggle op .html/.htm een dagelijkse wrijving weg.

Hoe het eruitziet

Bestaande Markdown-toggle (Code/Eye) hergebruikt voor HTML. Zelfde positie, zelfde shape, andere content erin.

Requirements

Negen requirements over drie concerns. R-IDs lopen door — R1–R3 routing, R4–R6 fidelity, R7–R9 update behavior.

Editor routing

3
  • R1EditorContainer routeert .html en .htm naar een Preview ⇄ Raw toggle, identiek aan de Markdown-UI.
  • R2Bestanden openen standaard in Preview. Raw gebruikt Monaco — zelfde code path als nu.
  • R3Toggle is per-tab en blijft staan zolang de tab leeft; opnieuw openen start weer in Preview.

Preview fidelity

3
  • R4Sandboxed iframe. Inline en externe <script> draaien. CSS, fonts, images laden.
  • R5Relatieve paden (./style.css, img/foo.png) resolven t.o.v. de file-directory — zoals een browser het bestand direct zou openen.
  • R6Console output, network errors en JS exceptions blijven binnen de iframe — geen crash of leak naar de host renderer.

Update behavior

3
  • R7Edits in Raw propageren naar Preview, debounced ~300ms na laatste keystroke.
  • R8Saven (Ctrl+S) levert geen extra re-render naast wat de debounce al deed.
  • R9Raw → Preview togglen toont de huidige buffer (incl. ongesavede edits), niet de disk-versie.

Update flow

Van keystroke tot rendered DOM in < 350ms. Geen save-trigger nodig.

Success Criteria

Hieraan herken je dat het ding doet wat het moet doen.

  • Gebruiker opent index.html, ziet meteen de gerenderde output, switcht naar Raw, edit een heading, en ziet de preview binnen ~300ms updaten — zonder te saven.
  • Een statische site met relatieve CSS & image references rendert hetzelfde als wanneer een browser het bestand direct opent.
  • Implementatie hergebruikt het toggle-patroon uit EditorContainer — geen nieuwe UI-affordances of editor-abstractie nodig.

Scope Boundaries

Wat zit erin, wat blijft eruit. Expliciete out-of-scope items maken planning veel sneller.

In scope

  • +.html en .htm bestanden
  • +Preview ⇄ Raw toggle in bestaande editor-tab
  • +Sandboxed iframe met scripts + asset loading
  • +Live, gedebounced updates (~300ms)
  • +Relatieve paden via <base href>

Bewust buiten scope

  • ×Aparte "preview" panel-architectuur — leeft in bestaand tab-systeem
  • ×.svg, .xhtml, andere markup formats
  • ×DevTools, element inspector, responsive-viewport toggle
  • ×Live reload van externe files (CSS in andere tab edit → geen auto-reload)
  • ×Eigen script-sandboxing bovenop iframe sandbox

Key Decisions

Vier keuzes met de rationale erachter — zodat een planner of code-reviewer ze niet opnieuw hoeft te maken.

1 Sandboxed iframe + base href

Closest to real browser behavior. Trade-off: previewed HTML draait JS, dus vijandige HTML kan proberen te phone-homen — maar de gebruiker draait toch al npm install en willekeurige terminal-commands op deze code. Threat model verandert niet.

2 Live debounced (~300ms)

Match met hoe Markdown's WYSIWYG nu al voelt. Scripts re-runnen bij elke preview rebuild — geaccepteerd, want R7 gaat over snelle feedback bij editing, niet over JS-state behouden.

3 Preview-first default

Consistent met Markdown die in WYSIWYG opent. Wie source-first wil, klikt de toggle één keer; we persisteren die voorkeur nog niet.

4 .html + .htm alleen

SVG ziet er aantrekkelijk uit maar voegt een as toe (ander content model, andere sizing) zonder duidelijke payoff. Defer naar later.
