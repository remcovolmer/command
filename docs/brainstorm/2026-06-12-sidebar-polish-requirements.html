Situatie
De sidebar is recent volledig herzien: urgency-only kleur, attention-rail, pinnable projects, plan-usage in de footer. Het werkt, maar er zijn kleine fricties blijven hangen.
Complicatie
Elke project-rij toont 4 hover-icons (visuele ruis). Alle worktrees worden getoond — ook die zonder draaiende Claude-sessie — wat projecten met veel branches vervuilt. Resolutie
Vier afgebakende ingrepen: hover-icons terug naar 2 (pin + verwijder achter rechtermuisknop), session-loze worktrees inklappen onder een "Show inactive worktrees"-toggle, de "Projects"-header weghalen, en de usage/token-tracker betrouwbaar zichtbaar maken onderaan — die verdwijnt nu stil zodra hij geen data heeft.

Requirements

1

Hover-icons van 4 naar 2; pin en verwijder achter rechtermuisknop

Op hover toont een project-rij alleen nog de twee creërende acties: New Chat (+) en — voor code-projecten — New Worktree (⑂). Pin/Unpin staat al in het rechtermuisknop-menu; Remove project verhuist daar ook naartoe (onder een separator, variant: 'destructive' — al ondersteund).

Waarom geen verlies: gepinde projecten staan al in hun eigen "Pinned"-sectie, dus de ster als pin-signaal is niet nodig. Verwijderen is een zeldzame, onomkeerbare actie — hoort niet als always-visible hover-icon.

Acceptatie

  • Hover op een code-project toont exact 2 action-icons; op een niet-code-project 1 (alleen New Chat).
  • De ster (pin) en de X (remove) zijn verdwenen uit de hover-acties.
  • Rechtermuisknop toont "Remove project" als laatste item, destructief gestyled, met bevestiging zoals nu (handleRemoveProject blijft de actie).
Raakt: src/components/Sidebar/SortableProjectItem.tsx (hover-acties + contextMenuItems). Geen nieuwe state. klein
2

Inactieve worktrees inklappen onder "Show inactive worktrees (N)"

Definitie: een worktree is actief als er ≥1 draaiende terminal/Claude-sessie op staat (getWorktreeTerminals(id).length > 0); anders inactief. Binnen een uitgeklapt project worden directe chats en actieve worktrees inline getoond; inactieve worktrees vallen onder een toggle-rij "Show inactive worktrees (N)", standaard ingeklapt.

Dit spiegelt het bestaande Active/Inactive-onderscheid op projectniveau, nu één laag dieper. De toggle-state wordt per project bewaard (zoals collapsedProjects / inactiveSectionCollapsed in de store).

Acceptatie

  • Een project met 4 worktrees waarvan 1 een sessie draait toont 1 worktree inline + "Show inactive worktrees (3)".
  • Klikken op de toggle vouwt de 3 inactieve worktrees uit/in; staat blijft per project bewaard over restart.
  • Geen toggle-rij als er 0 inactieve worktrees zijn.
  • Directe chats (terminals zonder worktree) worden niet ingeklapt — die draaien per definitie.
  • verplicht Hotkey toegevoegd (CLAUDE.md-eis): toggle inactieve worktrees voor het actieve project — voorgesteld Ctrl+Shift+J (niet bezet; Ctrl+Shift+I is al de project-inactive-toggle). Toevoegen aan DEFAULT_HOTKEY_CONFIG, handler in App.tsx, documenteren in de shortcuts-tabel.
Raakt: SortableProjectItem.tsx (split actieve/inactieve worktrees + toggle-rij), stores/projectStore.ts (nieuwe per-project collapse-state + action), utils/hotkeys.ts, App.tsx, CLAUDE.md (shortcuts-tabel). grootste item
3

"Projects"-header verwijderen

Verwijder het <h2>Projects</h2>-blok (Sidebar.tsx:385–390). De sectie-headers Pinned/Active/Inactive labelen de lijst al, en het onderscheid waar "Projects" ooit voor stond (tegenover workspaces) bestaat niet meer.

Acceptatie

  • Geen "PROJECTS"-label meer boven de lijst.
  • De afstand tussen de logo-header en de eerste sectie blijft bewust ogen — eventueel kleine padding-correctie waar het blok stond.
Raakt: src/components/Sidebar/Sidebar.tsx. triviaal
4

Usage/token-tracker betrouwbaar zichtbaar onderaan — nooit stil verdwijnen

De tracker is de plan-usage indicator uit #136 (UsageIndicator, footer): 5h-window + weekly utilization van Anthropic's OAuth /usage-endpoint. Wiring is compleet (hotkey App.tsx:419, IPC usage:update/usage:set-enabled, store-subscription op rehydration) en het endpoint geeft geldige data terug voor dit account (geverifieerd: 200 OK met five_hour+seven_day). Toch is hij onzichtbaar, want UsageIndicator returnt null zodra de toggle uit staat, er geen usageData is, óf status !== 'ok' — geen spoor, niet te onderscheiden van "verwijderd".

Wijziging (optie A — minimaal, gekozen)

  • Zolang de indicator aan staat maar (nog) geen bruikbare data heeft, render een gedimde "Usage · unavailable"-regel i.p.v. null — zo is hij altijd aanwezig onderaan en zelf-verklarend.
  • Zodra data binnenkomt wisselt de regel naar de normale balk. Dit dubbelt als diagnose: placeholder zichtbaar = data-flow-probleem; helemaal niks = toggle staat uit.
  • "Toggle uit" blijft bewust verbergen — daar is de toggle voor. Herstellen via Settings → General (GeneralSection.tsx:164) of Ctrl+Shift+U.
  • Geen herinrichting van de balk-styling in deze ronde (dat was optie B, niet gekozen).

Over "Ctrl+Shift+U doet niks": hotkey is wél gewireld. Meest waarschijnlijk slokt een gefocuste terminal de toets op, óf de toggle staat al aan maar usageData blijft null in de renderer. Optie A maakt dat laatste meteen zichtbaar.

Raakt: src/components/Sidebar/UsageIndicator.tsx (early-return vervangen door placeholder-render). Geen IPC/store-wijziging. klein

Buiten scope (non-goals)

  • Footer, usage-indicator, attention-rail en PR-badges blijven ongewijzigd.
  • De project-niveau Pinned/Active/Inactive-split blijft zoals hij is.
  • Geen per-sessie token-counter (tokens per chat) — Claude Code levert die data niet via de hooks/state die we lezen; buiten bereik.
  • De Recent Sessions-lijst (afgesloten sessies, hervatbaar) blijft in Project Overview — niet naar de sidebar verplaatst.
  • New Chat / New Worktree-gedrag verandert niet — alleen welke icons zichtbaar zijn.

Outstanding questions

VraagDefault als niet beantwoord
Geldt de "inactive worktrees"-toggle alleen voor het actieve project, of voor elk uitgeklapt project met inactieve worktrees? Elk uitgeklapt project — de logica hangt aan worktree-activiteit, niet aan project-active-ness. Consistenter en simpeler.
Verbergt het inklappen ook PR-status (conflict/#458) van die worktrees — is dat erg? Geaccepteerd. Eventueel later een rollup op de toggle-rij ("3 · 2 conflicts"). Nu niet bouwen (YAGNI).
Moet "toggle uit" toch een spoor achterlaten, of blijft uit = volledig weg? Uit blijft weg (doel van de toggle). Alleen de "aan-maar-geen-data"-staat wordt zichtbaar; herstellen via Settings → General of Ctrl+Shift+U.
Kleinste eerste stap: item 3 (de <h2>Projects</h2> weghalen) is één edit van ~5 regels en direct zichtbaar. Daarna item 4-A (UsageIndicator een zichtbare staat geven i.p.v. null), item 1 (icons verplaatsen, geen nieuwe state), en als laatste item 2 (de enige met nieuwe store-state + hotkey).
