Kern in één alinea

Situatie — na KD1 (terminal-first) is browser + annotatie de eerste Kestrel-feature, en de browser is intussen een echte <webview> (PR #152). De eis: drie manieren om vanuit die browser met de agent te praten — (1) selecteer + comment → snippet naar chat, (2) highlight + inline-edit → wijziging naar chat, (3) teken op de pagina + "voeg toe aan chat" → screenshot naar chat. Complicatie — de webview is bewust gehard (hardenWebviewPreferences: preload gestript, sandbox aan), dus je kunt geen bridge in de pagina injecteren om events omhoog te pushen. Resolutie — alle drie de modi werken via host-pull: Command's renderer roept executeJavaScript/capturePage aan op de webview-ref en houdt de interactie-UI in de eigen React-laag. Tekst gaat via terminal.write; de screenshot van modus 3 gaat via het klembord de chat in (Claude Code plakt afbeeldingen met Alt+V).

Product-autoriteit

Remco (eigenaar Command). Dit doc scopt wat; het hoe (overlay-techniek, executeJavaScript-payloads) is voor de planfase.

Open blocker vóór planning

Overlay van de interactie-UI over een <webview> — z-index/paint over een guest-view is een bekend Electron-pijnpunt; React-overlay vs. UI-in-de-pagina injecteren is te beslissen. Zie Outstanding Questions.

Product Contract

Summary

Bouw drie interactie-modi op de bestaande browser-webview. Command's renderer stuurt de webview aan via executeJavaScript/capturePage (host-pull); de mode-schakelaar en popover leven in Command's React-laag. Modus 1 stuurt de code-snippet van een selectie + comment als tekst naar de actieve Claude-chat; modus 2 laat je tekst ter plekke editen en stuurt de voor→na-wijziging als instructie; modus 3 laat je op de pagina tekenen en stuurt een screenshot van tekening + pagina als afbeelding via het klembord (Alt+V). De agent past de bron aan; de gebruiker ververst de pagina.

Problem Frame

Er is een gat tussen "iets zien dat niet klopt op een draaiende pagina" en "de agent dat laten fixen". De gebruiker moet de fout in woorden vangen, het juiste bestand/component benoemen en dat als prompt tikken — traag, foutgevoelig, en het breekt de visuele feedback-loop. Kestrel loste dit op door de aanwijzing letterlijk op de pagina te doen:

[31:37 / 32:41]

"This text is kind of small — I can highlight it and say: the labels are too small." → "It bumped up the SVG label size and we can just refresh."

Verschillende problemen vragen om verschillende gebaren: soms wil je iets beschrijven (comment), soms exact demonstreren hoe het moet (edit), soms alleen aanwijzen waar het mis is (tekening). Vandaar drie modi in plaats van één.

Key Decisions

KD-A · Host-pull, geen guest-bridge

Bouw op de bestaande, geharde webview via host-pull. webviewSecurity.hardenWebviewPreferences stript preload en zet sandbox/contextIsolation aan — bewust, als bridge-escape-verdediging. We verzwakken dat niet. In plaats daarvan roept Command's renderer executeJavaScript() en capturePage() aan op de webview-ref (host-side, overleeft de sandbox), en de interactie-UI leeft in Command's React-laag. Dit vervangt de oude aanpak (script in srcdoc dat postMessaget) die met de verwijderde HtmlPreview verdween.

KD-B · Alles naar de actieve Claude-chat

Elke modus stuurt naar de actieve chat via terminal.write (tekst) — nooit API/SDK. Modus 1 en 2 sturen tekst; modus 3 stuurt tekst die verwijst naar een opgeslagen screenshot-bestand (Claude leest de afbeelding via het pad). Consistent met KD1 en de gebruiker ziet de agent live werken.

KD-C · Source-mapping

Deterministisch voor lokale bestanden, agent-resolved voor dev-servers. Een lokaal bestand mapt naar zichzelf; een dev-server rendert component-output, dus Command stuurt rijke context mee (outerHTML/selector, DOM-pad, tekst, URL/route) en laat Claude — dat de repo heeft — de juiste component vinden.

KD-D · Modus 3: afbeelding via het klembord (Alt+V)

De screenshot gaat via het klembord de chat in. Claude Code plakt klembord-afbeeldingen met Alt+V (bevestigd werkend) — dat is een Claude Code-capability, geen Command-code. Command zet de capture op het OS-klembord (clipboard.writeImage, main-process, past bij het bestaande clipboard-over-IPC-patroon) en de afbeelding landt via Alt+V in de prompt. Dit vraagt alleen dat clipboard.writeImage wordt blootgelegd — geen binaire file-write of temp-bestand. Een opgeslagen-bestand-met-pad blijft de fallback.

Key Flows

F1 · Modus 1 — Aanwijzen & commentaar. Trigger: modus 1 actief. Gebruiker selecteert tekst/content in de webview → klikt "commentaar" → Command leest via executeJavaScript de selectie + outerHTML/selector + URL → toont een comment-invoer → gebruiker typt en verstuurt → Command schrijft snippet + comment + bron naar de chat → agent past aan → gebruiker ververst.

F2 · Modus 2 — Inline editen. Trigger: modus 2 actief. Gebruiker highlight tekst → Command zet dat element via executeJavaScript op contentEditable → gebruiker typt de gewenste tekst ter plekke → klikt "verstuur edit" → Command leest voor→na + locator terug → schrijft een edit-instructie naar de chat → agent past de bron aan → gebruiker ververst.

F3 · Modus 3 — Tekenen & snapshot. Trigger: modus 3 actief. Command injecteert een teken-canvas over de pagina (executeJavaScript) → gebruiker tekent → klikt "voeg toe aan chat" → Command roept capturePage aan (pagina + tekening) → zet de afbeelding op het klembord (clipboard.writeImage) → de afbeelding landt via Alt+V in de chat (Command triggert de paste of de gebruiker doet Alt+V) → agent bekijkt de afbeelding → gebruiker ververst.

Requirements

Overkoepelend

IDRequirement
R1De browser-toolbar heeft een mode-schakelaar tussen de drie modi (aanwijzen, editen, tekenen); één modus is tegelijk actief.
R2Elke modus stuurt het resultaat naar de actieve Claude-chat via terminal.write — niet via API/SDK.
R3Als er geen actieve chat van type claude is, meldt de UI dat en blokkeert verzending; het opgebouwde item blijft behouden.
R13De interactie draait via host-pull: Command roept executeJavaScript/capturePage aan op de webview-ref. Er wordt geen preload/bridge in de guest geïnjecteerd en de bestaande hardening blijft intact.

Modus 1 — Aanwijzen & commentaar

IDRequirement
R4De gebruiker kan tekst/content op de pagina selecteren en er een vrije opmerking aan koppelen.
R5Command legt de selectie vast als code-snippet (outerHTML/tekst) plus locator (selector/DOM-pad) en de bron-URL/route.
R6Het chatbericht bevat de snippet + de opmerking + de bron, zo dat de agent de juiste plek vindt.

Modus 2 — Inline editen

IDRequirement
R7De gebruiker kan een gehighlighte tekstpassage ter plekke in de pagina bewerken (via contentEditable, host-geïnitieerd).
R8Command legt de voor→na-tekst plus locator vast.
R9De wijziging gaat als edit-instructie ("verander X in Y op <locator>") naar de chat; de agent past de bron aan (werkt voor dev-server én lokaal bestand). De inline-edit in de pagina zelf is tijdelijk, geen bronwijziging.

Modus 3 — Tekenen & snapshot

IDRequirement
R10De gebruiker kan vrij over de pagina tekenen; een "voeg toe aan chat"-knop rondt de tekening af.
R11Command maakt een screenshot van pagina + tekening samen (capturePage op de guest, met de tekening in de guest gerenderd) en zet die op het klembord via clipboard.writeImage.
R12De afbeelding landt in de chat via Alt+V (Claude Code's klembord-image-paste); Command triggert de paste of de gebruiker doet Alt+V, met de bron-URL/route als begeleidende tekst.

Acceptance Examples

AE1 — Covers R4–R6 (modus 1).

Wanneer
op localhost:3000 selecteert de gebruiker een "Score"-label, kiest comment en typt "maak groter".
Dan
de chat krijgt de snippet (outerHTML) + selector + route + "maak groter", en Claude zoekt de component en past aan.

AE2 — Covers R7–R9 (modus 2).

Wanneer
de gebruiker highlight de kop "Reosultaten", corrigeert 'm inline naar "Resultaten" en verstuurt.
Dan
de chat krijgt "verander 'Reosultaten' → 'Resultaten' op <h2 op /report>", en Claude past de bron aan; de pagina zelf is niet permanent gewijzigd tot de agent klaar is.

AE3 — Covers R10–R12 (modus 3).

Wanneer
de gebruiker cirkelt een grafiek en een naastgelegen witruimte in, en klikt "voeg toe aan chat".
Dan
de PNG van pagina + cirkels komt op het klembord en landt via Alt+V in de chat (met de route als tekst erbij); Claude leest de afbeelding.

AE4 — Covers R3.

Wanneer
een modus wordt gebruikt terwijl er geen actieve claude-chat is.
Dan
de UI meldt dat er een chat nodig is en het item blijft behouden tot er een is.

Scope Boundaries

In scope

  • Drie modi op de bestaande webview (dev-server localhost + lokaal file://)
  • Host-pull architectuur; mode-schakelaar in de toolbar
  • clipboard.writeImage blootleggen (voor modus 3)
  • Verzenden naar de actieve Claude-chat

Uitgesteld / buiten scope

  • Externe/SaaS-URLs annoteren
  • Opgeslagen-bestand-met-pad als route voor modus 3 (fallback, niet primair)
  • Batchen van meerdere annotaties in één bericht (kan later; MVP = per actie versturen)
  • Annotatie-persistentie over sessies; verzwakken van de webview-hardening
Buiten de identiteit

Dit maakt van de browser geen visuele WYSIWYG-editor en vervangt de terminal niet. De modi zijn snelle input-methodes naar de chat.

Dependencies / Assumptions

  • Geverifieerd (na merge van #152): de browser is een <webview> met een ref in BrowserTab.tsx; webviewSecurity.hardenWebviewPreferences stript preload en zet sandbox/contextIsolation aan, partition gepind. terminal.write(id, data: string) bestaat (tekst-only). In Command zit vandaag géén image-clipboard-code (geen writeImage/readImage). Actieve browser-tab + URL zijn te vinden via activeContentTabId[activeTerminalId] → editorTabs[..] met type: 'browser'.
  • Bevestigd door gebruiker: Claude Code plakt klembord-afbeeldingen in de terminal via Alt+V. Dat is een Claude Code-capability (de CLI leest het OS-klembord), niet Command-code — Command hoeft de afbeelding alleen op het klembord te zetten.
  • Nieuw werk: (1) executeJavaScript op de webview-ref beschikbaar maken vanuit de renderer voor selectie/edit/canvas; (2) capturePage op de guest wiren; (3) clipboard.writeImage blootleggen via de main-process clipboard-module (klein; bestaand IPC-patroon).
  • Aanname (te bevestigen in plan): voor een dev-server vindt Claude de juiste component betrouwbaar genoeg uit outerHTML + selector + route; capturePage is bereikbaar op de guest-webContents; het Alt+V-paste-signaal kan programmatisch getriggerd worden (anders drukt de gebruiker Alt+V).

Outstanding Questions

Op te lossen vóór planning

  • Overlay-strategie: leeft de mode-UI (popover, comment-invoer) als React-overlay boven de <webview>, of injecteren we de UI in de guest via executeJavaScript? Overlay over een guest-view is een bekend z-index/paint-pijnpunt; dit raakt alle drie de modi (R13/R1).
  • Selectie-UX: host-pull betekent dat een selectie op een trigger wordt uitgelezen (knop/hotkey). Accepteren we die extra klik, of gebruiken we het console-message-kanaal om selecties toch event-gedreven te laten verschijnen (zonder de hardening te verzwakken)?

Uit te zoeken tijdens planning

  • Locator-formaat (R5/R8): welke combinatie van selector, DOM-pad, tekst en route vindt Claude het betrouwbaarst terug; benutten we dev-mode source-attributen indien aanwezig?
  • capturePage-scope (R11): volledige pagina of alleen viewport; hoe de tekening in de guest renderen zodat 'ie in de capture zit.
  • Alt+V-trigger (R12): kan Command de paste programmatisch triggeren (byte-sequence naar de PTY), of drukt de gebruiker zelf Alt+V — en zet clipboard.writeImage de NativeImage zo neer dat Claude Code 'm oppikt?
