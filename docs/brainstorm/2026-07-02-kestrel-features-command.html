Kern in één alinea

Situatie — Kestrel is in ~48 uur gebouwd door Jack, senior editor (géén engineer) bij Every, omdat hij "een Codex voor Claude Code" wilde: een GUI-laag boven de agent in plaats van een terminal. Complicatie — de demo bevat drie ideeën die Command raken (Sweep-loops op documenten, browser-annotatie die de agent uitvoert, en Codex + Claude Code náást elkaar) plus een reeks kleinere mechanismes die makkelijk over het hoofd te zien zijn. Resolutie — het grootste deel van de infrastructuur voor het krachtigste idee (Sweep) zit al in Command; de missende stukken zijn klein en concreet. Multi-backend is echt greenfield. Dit document inventariseert alles en zet een aanbevolen prioritering neer — de keuze "welke pakken we op" is de volgende stap.

Product-autoriteit

Remco (eigenaar Command). Geen externe stakeholders. Prioritering wordt in de vervolgstap samen bepaald; dit doc is de menukaart, geen commitment.

Kernbeslissing — beslist 2026-07-02

Command blijft terminal-first zoals het nu werkt. Een GUI/chat-shell wordt pas overwogen als we tegen iets aanlopen dat niet anders op te lossen is — en dán via wrap-de-CLI, nooit via de SDK/API. Reden: te veel onderhoud voor een tool die nu alleen Remco gebruikt. Zie Kernbeslissing.

Wat is Kestrel

Kestrel ("torenvalk") is een zelfgebouwde desktop-app die Jack omschrijft als een vogel-compagnon: "this little bird companion that kind of sits on your shoulder and then goes and does things for you" (30:33). Het is expliciet gepositioneerd als de Codex-desktopervaring, maar dan draaiend op Claude Code:

[30:01]

"I'm usually in Codex and I was like: oh, I wish I had a Codex for Claude Code. And then once Fable came out I was like: maybe I can build one."

De belangrijkste ontwerpgedachte is dat het géén dev-tool is. Codex en Claude Code zijn gemaakt voor developers; Jack wilde iets dat "the wide range of tasks that I do" ondersteunt — schrijven, research, markdown (40:39). Zijn stelling: "AI turns everything into user experience" (32:09) — je bouwt precies de interactie en het gevoel dat je wilt, in plaats van je te voegen naar een gespecialiseerd product. Kestrel bestaat uit vier bouwstenen: een ingebouwde browser, annotatie op een surface, Sweep (agent-loops), en het draaien op bestaande Claude- én Codex-abonnementen tegelijk.

Volledige analyse van de demo (29:00–52:00)

1. Positionering & architectuur — een laag, geen terminal

Kestrel toont nergens een terminal. Het is een GUI die de agent aanstuurt. Het draait bovendien op meerdere backends tegelijk en erft de connectors (MCP) van die accounts:

[32:41]

"This is running off of my existing Claude and Codex subscriptions, and therefore it has all my existing connectors that I have linked up to those accounts."

Dit is precies het punt dat je zelf signaleerde: Kestrel gebruikt Claude Code niet als terminal, maar als een aanstuurbare motor eronder. Vergelijkbaar met Conductor, dat Jack ook noemt als inspiratie (32:41).

2. Sweep — de kern-innovatie

Sweep is een agent die een surface (een webpagina óf een markdown-document) in een loop bewaakt en erop handelt. Het idee komt van Dan's "inbox sweep" (33:19). De mechaniek bestaat uit zes onderdelen die elk apart de moeite waard zijn:

  • Brief per surface. De eerste keer dat je een surface sweep't, vraagt Kestrel wat je wilt: "the first time around, it actually says: I need to know what you want me to do" (43:11). Die brief wordt opgeslagen ("the brief saved", 44:55) en hoort bij dát document/die pagina.
  • Marker-gedreven capture. De trigger is schrijver-native syntax, geen aparte takenlijst-UI: "anytime that I have TKs or things in brackets, it will as part of its sweep pick up those indicators" (34:26). TK en [...] zijn wat een schrijver tóch al gebruikt voor "moet nog gebeuren".
  • Diff-gating. Sweep vergelijkt met de vorige versie en doet niets als er niets veranderd is: "it actually compares it against the previous version" → "it says nothing new" (37:00). Alleen op veranderingen wordt gehandeld.
  • Done-migratie. Afgeronde items schuiven naar een done-sectie: "once it's completed with the item, it moves it into the done section... this is everything I've done on this app" (37:00). Dat levert een audit-trail op van alles wat de agent deed.
  • Auto-execute vs. bevestigen. Een toggle: "I can also have it just auto-execute — every sweep it'll just go and execute" (39:37).
  • Self-building loop. Jack richtte Sweep op de repo van Kestrel zélf, met een taken-document en de instructie: "every sweep, if there's an item that feels manageable and doesn't require too much work, just go ahead and do it" (36:28). De app bouwt zichzelf op laaghangend fruit.

3. Browser + annotatie — highlighten en de agent doet het werk

Een van Jack's favoriete features uit Codex/Cursor: annoteren op een gerenderde pagina en de agent voert het uit.

[31:05]

"One of them is the ability to annotate on the web page and have it do the work... I can highlight it and say: the labels are too small." → "it bumped up the SVG label size and we can just refresh."

De feedback-loop is visueel en ruimtelijk: zie iets → markeer het op de surface → comment → agent past aan → refresh. Geen prompt tikken over een abstract bestand.

4. Interactiemodel — polling vs. on-demand, en twee soorten agents

Dan levert live de belangrijkste ontwerpkritiek: hij wil géén interval-polling, maar event-triggering.

[41:40 / 44:22]

Dan: "For sweep, I'd prefer that if I type something it steers it — sends a message — rather than the chat having to pull... I want it to not be on a sweep, I want it to be on demand. Instead of it polling, I want it to be activated."

Jack's antwoord verklapt de reden: polling is duur. "Right now I don't have that built in yet, partially because it's very costly with Fable to do that" (44:22). Verder benoemt Every twee modellen van samenwerken (51:00):

Delegate-agent

"One agent that you delegate everything to and just don't care about being in a document live." Async hand-off. Vereist betrouwbaarheid.

Pair-agent

"The other that you're working with simultaneously while you're in the same file." Real-time. Vereist snelheid — Fable op low effort is snel genoeg.

Hieruit volgt de effort-dial als cost/speed-hendel: low effort = goedkoop + snel genoeg voor real-time; hoger voor moeilijkere taken (46:26, 51:33).

5. Ambient / life-integration

Het onderliggende werkmodel is asynchroon en verweven met het leven, niet "voor een chatvenster zitten". Jack heeft een peuter en wil ideeën vangen zonder te chatten:

[37:32 / 49:00]

"Wednesday morning I was doing laundry and in between folding clothes I would think of things to add — I'd just add it and I had the sweep running... I want to capture my thoughts, monologue them into notes, and then just have it automatically act on that note on a regular interval."

De evolutie die hij schetst: twee weken eerder liet hij Claude 's nachts zijn notitiebestand scannen en ideeën ordenen; nu kan het niet alleen ordenen maar ook taken uitvoeren (49:30–50:00).

Punten die je nog miste

Je noemde Sweep, Codex+Claude Code, de laag-in-plaats-van-terminal en de browser-met-comments. Dat zijn de grote drie. Dit zijn de subtielere mechanismes die minstens zo bepalend zijn:

  • Brief per surface — instructies zitten vást aan een document/pagina, niet aan een chat. Dit is het verschil tussen "een prompt" en "een staande opdracht op een bestand".
  • Diff-gating ("nothing new") — de loop doet niets tenzij het bestand veranderde. Cruciaal tegen kosten en ruis.
  • Done-migratie / audit-trail — afgeronde items verhuizen zichtbaar naar "done". Command doet dit vandaag al voor TASKS.md.
  • Self-building loop — de app op zijn eigen repo richten en laaghangend fruit autonoom laten oppakken.
  • Polling vs. event-triggering + de kostendriver — Dan's kritiek is het belangrijkste architectuur-inzicht. En hier ligt Command juist vóór: file-change-triggers zijn al event-based, niet polling.
  • Auto-execute vs. bevestigen als expliciete toggle.
  • Twee-agent-model (delegate vs pair) als productfilosofie, met betrouwbaarheid vs. snelheid als de twee vereisten.
  • Effort-dial als eerste-klas cost/speed-hendel.
  • Connector-erfenis — MCP-connectors komen automatisch mee van de gekoppelde accounts.
  • Ambient capture — inbox-note die op interval wordt verwerkt; het echte gebruiksmodel achter Sweep.

Kernbeslissing: terminal-first vs. GUI-layer

Command is vandaag terminal-first: TerminalManager spawnt een PTY en draait claude (met --resume en auto-mode). Kestrel laat de terminal helemaal weg. Voor Command is dit primair een kosten- en onderhoudsvraag, geen esthetische. Twee feiten uit de scan bepalen het antwoord: (1) Command praat niet met de API maar draait de claude CLI als subprocess — dat houdt alles op het flat abonnement, niet op metered API-billing; (2) de rauwe terminal toont de CLI-stream 1-op-1, dus élke nieuwe Claude Code-feature (user questions, subagents, skills) verschijnt vanzelf — nul onderhoud.

Beslist 2026-07-02 — KD1 resolved

Command blijft voorlopig terminal-first zoals het nu werkt. Een GUI/chat-shell wordt pas overwogen als we tegen iets aanlopen dat niet op een andere manier op te lossen is. Reden: een geparste chat-UI creëert een feature-treadmill (elke nieuwe interactieve primitive moet je bijhouden) — te veel onderhoud voor een tool die nu alleen Remco gebruikt.

Nu · gekozen

Terminal zoals nu gekozen

Status quo: PTY + claude CLI. Sweep en annotatie kunnen als additieve surfaces bovenop de bestaande Automation/FileWatcher/Editor/Browser-infra — die vervangen de terminal niet, dus ze staan los van deze keuze.

Later · alleen als nodig

GUI-shell via wrap-de-CLI uitgesteld

Als de switch ooit komt: render de CLI-output in eigen UI mét de terminal als source-of-truth-fallback. Blijft op het abonnement, geen treadmill. Nooit via SDK/API (= metered + herbouw). Trigger: een concreet probleem dat niet anders op te lossen is.

Impact op de feature-lijst

FC16 (terminal-loze modus) blijft uitgesteld. FC10 (echte webview) komt juist in scope: de ingebouwde browser wordt geconsolideerd tot enige page-surface en van iframe naar webview geüpgraded — dat is de enabler voor FC9 (annotatie op dev-server + lokaal bestand). Sweep-features (FC1–FC8) draaien op de automation-infra (claude -p) en blijven los op tafel.

Sweep is al grotendeels gebouwd

Het sterkste, verrassendste inzicht uit de capaciteitsscan: de infrastructuur voor Sweep zit voor het overgrote deel al in Command. Vier van de bouwstenen bestaan, drie ontbreken — en de missende zijn klein en scherp afgebakend.

Consequentie voor de eerste stap

De kleinste concrete eerste stap is één verticale plak: een marker-getriggerde sweep op één markdown-bestand — file-change-trigger (bestaat) + een per-bestand brief (nieuw, klein) + claude -p-runner (bestaat) + done-migratie via het TaskService-patroon (bestaat). Dat is FC1 + FC2 + FC4 + FC6 samen als één dunne slice, bouwbaar zonder de terminal-architectuur te raken.

Aanbevolen prioritering

De logica: begin waar Command al 65% klaar is (Sweep-MVP), bewijs de waarde met een testcase, en stel de dure architectuur-keuzes (multi-backend, echte webview) uit tot na dat bewijs. Prioritering is een POV — de definitieve keuze maken we samen in de vervolgstap.

Tier 1 — Quick wins: de Sweep-MVP

FC1, FC2, FC3, FC4, FC6 (+ FC14, FC5 als kleine bijvangst). Eén verticale plak: marker-getriggerde sweep op één markdown-bestand, gebouwd op FileWatcher + Automation + TaskService + claude -p. Levert het kern-Kestrel-gevoel zonder de terminal-architectuur te raken. Dit is de aanbevolen eerste stap.

Tier 2 — Strategische bets (na bewijs)

FC9 (annotatie-overlay — de meest "wow" feature), FC15 (delegate/pair expliciet maken), FC7 (self-building loop), FC17 (ambient inbox), en FC11/FC16 als de grote richtingkeuzes. FC11 (multi-backend) en FC16 (terminal-loze modus) zijn de twee die het meest raken aan de identiteit van Command — bewust apart houden.

Tier 3 — Groter / later of overslaan

FC10 (echte webview i.p.v. iframe — flink werk, alleen als annotatie op live-web echt nodig blijkt), FC12 (connector-erfenis — afhankelijk van multi-backend), FC13 (route-by-fit — hangt aan FC11). FC8 is trivia; meenemen als het toevallig langskomt.

Openstaande vragen

Uit te zoeken tijdens planning

  • Is de effort-dial (FC14) beschikbaar via de Claude CLI-flags die Command gebruikt, of vergt dat SDK-werk? (Verify vóór inplannen.)
  • Multi-backend (FC11): is er een realistische Codex-CLI/headless-modus die net als claude -p aanstuurbaar is? (Onbevestigde aanname — checken.)
  • Diff-gating (FC3): FileWatcher levert events, maar bewaren we een snapshot per surface om "nothing new" te bepalen, of vertrouwen we op change-events alleen?
