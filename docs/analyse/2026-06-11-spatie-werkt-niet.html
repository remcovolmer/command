Spatie valt weg omdat Windows' tekstsuggestie/IME-laag de spatietoets als keyCode 229 aflevert — en xterm.js die per ontwerp niet als toetsaanslag verwerkt

Root cause analyse · Command Center · 11 juni 2026 · branch fix/spatie-werkt-niet

Situatie

In Claude-chats (xterm.js terminals) valt na verloop van tijd alléén de spatiebalk uit. Alle andere toetsen blijven werken.

Complicatie

Geen zichtbaar patroon; wisselen naar een ander venster en terug herstelt het. Eigen code (hotkeys, key handlers) behandelt spatie nergens apart.

Resolutie

De oorzaak zit in de interactie tussen Windows' IME/tekstsuggestie-laag en xterm.js' compositie-afhandeling — niet in Command Center-code. Fix: omgevingsinstelling én een app-level vangnet.

Waarom alléén de spatie uitvalt: twee verschillende routes door xterm.js

Zodra een IME-achtige laag (dead keys bij US-International, of Windows 11 "tekstsuggesties bij fysiek toetsenbord") actief wordt op het verborgen invoerveld van xterm, krijgen toetsaanslagen keyCode 229 ("door IME verwerkt"). xterm.js verwerkt die nooit als toets — het kijkt alleen achteraf of er tekst in het verborgen textarea is verschenen:

// @xterm/xterm 5.5.0 — CompositionHelper.keydown (gedeminificeerd)
keydown(e) {
  if (this._isComposing || this._isSendingComposition) {
    if (e.keyCode === 229) return false;            // IME-toets: xterm negeert hem
    if (e.keyCode === 16 || 17 || 18) return false;
    this._finalizeComposition(false);
  }
  return e.keyCode !== 229 ||
    (this._handleAnyTextareaChanges(), false);      // 229 buiten compositie: alleen textarea-diff
}
Letter (bijv. "a") tijdens IME-laag → komt aan
keydown met keyCode 229 — xterm verwerkt de toets niet
↓
IME voegt "a" toe aan het verborgen textarea
↓
_handleAnyTextareaChanges() ziet het verschil en stuurt "a" alsnog naar de PTY ✓
Spatie tijdens IME-laag → verdwijnt
keydown met keyCode 229 — xterm verwerkt de toets niet
↓
IME consumeert de spatie als commit/accepteer-actie en zet géén " " in het textarea
↓
Textarea-diff is leeg → er wordt niets naar de PTY gestuurd ✗

Spatie is voor IME-/suggestielagen een besturingstoets (suggestie accepteren, dead-key committen), geen tekstinvoer. Letters zijn dat niet — precies de scheidslijn die je waarneemt.

De openstaande schakel verifiëren kost 1 minuut. Check: (1) Windows-instellingen → Tijd en taal → Typen → "Tekstsuggesties weergeven tijdens het typen op het fysieke toetsenbord" — staat die aan? (2) Gebruik je de toetsenbordindeling Verenigde Staten (internationaal) (dead keys op ' " ` ~ ^)? En de volgende keer dat het optreedt: plak via Ctrl+V een spatie — komt die wél aan, dan is de keten definitief bevestigd (plakken omzeilt het toetsenpad volledig).

Fix-opties: omgeving aanpakken, app-vangnet, of beide

A — Omgevingsfix (geen code)

  • Tekstsuggesties voor fysiek toetsenbord uitzetten en/of wisselen naar layout zonder dead keys
  • Lost de oorzaak op bij de bron; ook effectief buiten Command Center
  • Helpt andere gebruikers met dezelfde setup niet

B — App-level vangnet (code)

  • In useXtermInstance.ts: detecteer spatie-keydown (event.code === 'Space', geen modifiers) die als keyCode 229 binnenkomt en waarvoor géén data bij de PTY aankomt → injecteer zelf ' '
  • Werkt voor iedere gebruiker, onafhankelijk van Windows-instellingen
  • Vergt zorg rond echte CJK-compositie (daar mág spatie geconsumeerd worden); testbaar als unit-test op de watchdog-logica
Advies: eerst de omgevingscheck (bevestigt de keten en is mogelijk direct de oplossing), daarna beslissen of het app-vangnet de moeite waard is voor robuustheid richting andere gebruikers.
