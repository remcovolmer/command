Goal Capsule

Objective. Give Command the same ambient plan-usage awareness for Codex that it already has for Claude — a second labeled bar in the sidebar footer — fed from Codex's locally-persisted rate-limit snapshots rather than a network endpoint.

Product authority. Remco (Command owner). Both product forks were resolved in the brainstorm dialogue.

Open blockers. None. Feasibility and the data shape were verified live this session against ~/.codex. Remaining choices are implementation-level (see Outstanding Questions) and belong to planning.

Product Contract

Summary

Add a second usage row to the sidebar footer for Codex, mirroring the existing Claude bar (percentage + reset time, threshold colors, hover detail). Its data is read locally from the newest Codex session rollout file — ~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl — by taking the last rate_limits block off its token_count events. No network call, no token, nothing but derived numbers crossing IPC. Each provider's row appears only when that provider's credentials exist, so a Claude-only machine looks exactly like today. Codex reuses Claude's two-state model (ok / unavailable) unchanged; a window whose resets_at is already in the past is treated as unavailable.

Problem Frame

Command already runs Codex chats as a first-class agent alongside Claude (codex binary, hooks in ~/.codex/hooks.json). Codex's ChatGPT-plan limits bite the same way Claude's do — a 5-hour window and a weekly cap — and you only discover the wall when Codex refuses mid-session. The Claude usage indicator shipped that ambient warning for Claude alone, leaving Codex blind.

OpenAI exposes no clean usage endpoint equivalent to Anthropic's oauth/usage. But the Codex CLI persists a rate-limit snapshot into every session rollout after each turn. Verified this session: rollout files carry token_count events with a rate_limits block — primary and secondary, each { used_percent, window_minutes, resets_at }, plus credits and plan_type. That local snapshot is exactly what the Codex CLI itself shows, and it is enough to drive the bar.

Key Decisions

KD1
Local rollout-file read, not a network endpoint.

OpenAI has no clean usage endpoint to mirror Anthropic's. The Codex CLI persists its rate-limit snapshot into session rollout files after every turn, and that snapshot is what the CLI's own status shows. Reading it is cheaper and simpler than the Claude path — no token handling, no auth-expiry classification, no network failure modes.

KD2
Two labeled rows in the footer, each shown only when its credentials exist.

Chosen over a single toggled bar or side-by-side segments: labeled stacked rows read cleanest and scale to a third agent later. Conditional visibility keeps a Claude-only machine identical to today and avoids a permanent "n/a" Codex row for people who never use Codex.

KD3
Codex reuses Claude's two-state model; a rolled-over window is unavailable.

No new "stale" UI state. Usage is monotonic within a window, so an old-but-not-yet-reset snapshot is still correct — the number only goes wrong once resets_at passes. That resets_at < now check is Codex's structural equivalent of Claude's failed-poll → unavailable, collapsing into the muted placeholder that already exists. Symmetric with Claude, zero extra states.

KD4
Windows are keyed by window_minutes, not by fixed field names.

Codex reports generic primary/secondary windows. Verified values are 300 (5h) and 10080 (weekly) — the same two horizons Claude has — but the label is derived from the minutes rather than hardcoded, and a null/absent secondary (seen transiently at session start) is tolerated. This also nudges the shared UsageData shape toward a provider + generic-window-list model instead of Claude-specific fiveHour/sevenDay keys.

KD5
One indicator controls both providers in v1.

The existing showUsageIndicator setting and Ctrl+Shift+U hotkey toggle the whole indicator (both rows) together. A per-provider toggle is deferred — no evidence it's wanted yet.

Requirements

Codex data source
IDRequirement
R1A main-process reader locates the most-recently-modified rollout-*.jsonl under ~/.codex/sessions/ and extracts the rate_limits block from the last token_count event in it.
R2The Codex bar renders the shortest available window (percentage + reset time) and colors by the highest utilization across all present windows — the same "closest-to-binding wins" rule as the Claude bar.
R3Each window's label is derived from its window_minutes (e.g. 300→"5h", 10080→"wk"); a null or absent secondary is tolerated without error.
R4Reading is local only: no network request, no credential token, and only derived numbers cross the IPC boundary.
Freshness & availability (symmetric with Claude)
IDRequirement
R5Codex usage has exactly two states, matching Claude: ok (render the bar) and unavailable (render the existing muted placeholder). No third "stale" state.
R6A snapshot whose relevant window resets_at is in the past counts as unavailable — the pre-reset percentage no longer reflects the current window.
R7Missing/unparseable ~/.codex/auth.json, or no readable rollout with a rate_limits block, → unavailable; the row reappears automatically once a fresh snapshot lands. No re-auth flow.
R8Transient read errors keep the last-good value (never flash empty on one failed read); an update is pushed only when the payload actually changed.
Sidebar presentation
IDRequirement
R9The footer shows one labeled row per provider (Claude / Codex). A row appears only when that provider's credentials exist (~/.claude/.credentials.json / ~/.codex/auth.json). With only Claude present, the footer is identical to today.
R10Each row keeps the existing bar + percent + reset-time affordance and hover popover. The Codex popover shows its windows (derived labels), plan_type, and the credits balance when has_credits is true.
Control & conventions
IDRequirement
R11The existing showUsageIndicator setting and Ctrl+Shift+U hotkey control both rows together; no separate per-provider toggle in v1.
R12New decision logic — rollout selection, rate_limits parse, window-label derivation, the resets_at staleness check, and level→color mapping — is extracted as pure exported functions and unit-tested with Vitest, matching the Claude service.

Acceptance Examples

  • AE1 · Covers R6. Newest rollout's driving window resets_at is yesterday → the Codex row shows the muted placeholder, not the stale percentage.
  • AE2 · Covers R6, R8. resets_at is in the future and no newer rollout exists → the row keeps showing the last snapshot's percentage unchanged (still correct — usage only rises within a window).
  • AE3 · Covers R9. Only ~/.claude/.credentials.json exists → the footer shows the Claude row only; no Codex row, no placeholder for it. Matches today's behavior.
  • AE4 · Covers R9. Both credential files exist → two stacked labeled rows.
  • AE5 · Covers R2, R3. Snapshot with primary.window_minutes=300 (37%) and secondary.window_minutes=10080 (62%) → the bar shows the 5h window, colored by the 62% weekly, with a "wk 62%" hint. A snapshot with secondary=null → single 5h line, no error.

Scope Boundaries

Outside this feature
  • ChatGPT web-chat usage. No local data source exists for it. "Codex usage" here means the Codex CLI's plan quota — which is the ChatGPT-plan quota — not the web chat.
  • Per-chat / per-token cost metrics. A different feature (the rollout token_count also carries token totals, but ambient per-chat cost is out of scope here).
  • Pi usage. The third agent exposes no rate-limit data; not in this pass.
  • Live probe calls. Forcing a throwaway API call just to read fresh rate-limit headers — rejected (costs tokens, hacky, no clean endpoint).
Deferred for later
  • A separate per-provider toggle, if hiding one row turns out to be wanted.
  • macOS credential/rollout path differences — the row hides silently if the files are absent (matches R7); a platform-specific reader is follow-up work.

Dependencies & Assumptions

Verified this session
  • ✓ Codex CLI writes rate_limits into session rollout .jsonl files (live rollout inspected).
  • ✓ used_percent is on a 0–100 scale (values up to 62.0 observed) — no fraction conversion needed, unlike the Claude used_credits-cents case.
  • ✓ Two windows exist: window_minutes 300 (5h) and 10080 (weekly). resets_at is a unix epoch in seconds (Claude used ISO strings — parse accordingly).
  • ✓ ~/.codex/auth.json present with auth_mode: "chatgpt" and tokens.access_token; no API key needed for the file-read path.
Assumptions to confirm in planning
  • The rate-limit snapshot is account-global and identical across concurrent sessions, so the newest rollout file is the freshest source. Plausible but not proven across all Codex versions.
  • The newest rollout always contains at least one rate_limits block; if a just-started session has none yet, the reader falls back to the next-newest (see Outstanding Questions).

Outstanding Questions

Resolve before planning

None — both product forks (UI shape, staleness handling) are resolved above.

Deferred to planning

  • Architecture. Extend the existing UsageService to also read Codex, or add a sibling reader — and generalize the UsageData / usage:update payload to carry a provider identity plus a generic window list.
  • Refresh mechanism. Reuse the focus-aware 60s/300s poll cadence for the file read, or watch ~/.codex/sessions via the existing FileWatcherService. Lean poll-cadence for v1.
  • Rollout selection edge case. Whether to read only the single newest file or scan the few newest to skip an aborted session that has no rate_limits event yet.
