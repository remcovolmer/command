Problem Frame

.html opent vandaag in Monaco als ruwe source. De Markdown-branch in src/components/Editor/EditorContainer.tsx heeft het toggle-patroon al uitgewerkt — Code/Eye-buttons die Monaco vervangen door Milkdown. We knippen dat patroon, plakken het voor HTML.

Requirements Trace

Negen requirements uit de origin doorgetrokken naar dit plan. Elke unit hieronder linkt expliciet terug.

R1Routing: .html/.htm krijgen Preview ⇄ Raw toggle
R2Preview is default-mode; Raw via bestaand Monaco-pad
R3Toggle per-tab, niet persistent across reopen
R4Sandboxed iframe; scripts, CSS, fonts, images laden
R5Relatieve paden resolven t.o.v. file-directory
R6Iframe errors blijven binnen iframe — geen host crash
R7Raw → Preview debounce ~300ms
R8Ctrl+S triggert geen extra re-render
R9Toggle naar Preview toont live buffer, niet disk

High-Level Technical Design

Structuur op één blik. Volgorde: file-router → owner → preview-iframe.

Note: deze schets illustreert de richting voor review — geen implementatie-spec. Een implementing agent gebruikt het als context, niet als code-template. 
EditorContainer  (file router)
   │
   ├─ .md           ──►  MarkdownEditor   (existing, unchanged)
   │
   ├─ .html | .htm  ──►  HtmlEditor       (NEW)
   │                       │
   │                       ├─ owns content (string)
   │                       ├─ Raw     ──► Monaco (mounted always)
   │                       └─ Preview ──► HtmlPreview (NEW)
   │                                          │
   │                                          └─ <iframe sandbox="allow-scripts"
   │                                                     srcdoc={withBaseHref(content, fileDir)} />
   │
   └─ *             ──►  CodeEditor      (existing, unchanged)

   Monaco.onChange ──► setContent ──► debounce(300ms) ──► iframe.srcdoc

Key Technical Decisions

Zes keuzes met de rationale erachter — reviewer hoeft ze niet opnieuw te maken.

1 Nieuwe HtmlEditor, geen refactor van CodeEditor

Geen blast radius op andere extensies; matcht MarkdownEditor's pattern (ook eigen owner). Kost wat dupe-lifecycle, levert nul risico op bestaande paden.

2 srcdoc + base href, niet src="file://"

srcdoc kan de live buffer renderen (R9); src= kan dat niet. Base href dekt R5. Risico: Electron security flags — eerste smoke-test in U2.

3 Sandbox: alleen allow-scripts

Minimum voor script-execution (R4). allow-same-origin bewust uit — null-origin permitteert assets via base href maar blokkeert cookie/storage access. allow-top-navigation uit — geen browser-jacks.

4 Debounce in component, geen shared util

300ms timer op Monaco's onChange. Niemand anders heeft dezelfde need; geen reden om dit als shared hook te abstraheren.

5 Preview-first default, niet user-persisted

Match met Markdown WYSIWYG-default. Last-used-mode persistence is een follow-up als gebruikers erom vragen.

6 .htm mee in EXT_TO_LANGUAGE

Beide zijn real-world HTML. Eén tabel-entry. Zero risk.

Open Questions

Wat is hier opgelost; wat moet de implementing agent ontdekken.

Resolved tijdens planning

  • Moet de markdown-branch een kind-enum worden?
    Nee. Component is ~95 lines; derde branch kost ~15 lines. Switch-refactor is churn zonder simplificatie.
  • Moet CodeEditor controllable worden om de Monaco-instance te delen met de preview?
    Nee. HtmlEditor mount eigen Monaco direct. Duplication is contained en voorspelbaar.

Deferred naar implementatie

  • Werkt srcdoc + <base href="file://"> onder Electron's default webSecurity: true?
    Test in U2 wire-up. Fallback bij blokkade: (a) Electron protocol-handler die file-dir served, of (b) inline-resolve assets als data-URLs. Beide zwaarder — alleen invoeren als simpel pad faalt.
  • Gedraagt dev (Vite localhost) zich anders dan production (file://)?
    Beide testen in U2/U3. Als dev breekt maar prod werkt: documenteer de dev-limitatie i.p.v. extra infrastructure bouwen.
  • Iframe key={debouncedContent} voor remount, of srcdoc in-place updaten?
    Eerst srcdoc-mutation. Bij script-misgedrag op update → switch naar remount. Eén regel verschil.

Risks & Mitigations

Wat kan fout gaan; wat is het plan.

RiskMitigation
srcdoc + file:// base geblokkeerd door Electron webSecurity U2-wire-up test dit eerst. Fallbacks (protocol-handler, data-URL inlining) gedocumenteerd. Geen investering vóór bevestigde blokkade.
Vite dev-server gedraagt zich anders dan production file:// Beide environments testen in U2/U3. Dev-limitatie documenteren i.p.v. dev-only infrastructure bouwen.
Vijandige HTML in workspace runt JS in preview Geaccepteerd. Zelfde threat model als npm install en terminal-commands die de app al draait. Geen mitigatie in v1.
Zware/looping JS bevriest preview-iframe Renderer-proces is niet de iframe; toggle naar Raw werkt nog. Geen timeout-kill in v1.
Bundle size groeit door nieuw editor-module Verwaarloosbaar — Monaco + React zijn al geladen; alleen ~200 lines nieuwe component-code.
