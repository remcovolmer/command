Goal Capsule

Doel: verberg de zichtbare scrollbar alleen op de chat-terminal (waar Claude Code draait) zodat de terminal-inhoud hem niet langer overlapt, terwijl scrollen via muiswiel/trackpad blijft werken. De sidecar-terminals blijven ongewijzigd.

Gezag: bevestigde gebruikerskeuze (alleen chat-terminal; scrollbar verbergen, niet stylen) is leidend. Repo-conventies en het bestaande terminal-scrollgedrag (PR #38) overrulen losse implementatiedetails. Stop-condities: klaar wanneer de scrollbar op de chat-terminal onzichtbaar is, de sidecar-terminal zijn scrollbar behoudt, wiel-scroll blijft werken en de bestaande testsuite groen blijft. Stop en meld als de wijziging onverwacht het scroll-naar-onder-gedrag of de breedte-fit raakt, of de sidecar toch beïnvloedt.

Product Contract

Summary

De Claude Code-chatterminal toont een geforceerde, ongestylede native scrollbar die door de terminal-inhoud wordt overlapt. De chat- en sidecar-terminals delen vandaag exact dezelfde class terminal-container, dus een regelwijziging op die class raakt beide. Geef daarom de chat-terminal een eigen marker-class (chat-terminal) en verberg de scrollbar gescoped daarop — exact het patroon dat de app al gebruikt in .scroll-hidden. Wiel-/trackpad-scrollen blijft werken; het verwijderen van de scrollbar-goot geeft de TUI de volle breedte en lost de overlap aan de rechterrand op. De sidecar-terminals houden hun huidige scrollbar.

Problem Frame

Op de gedeelde regel .terminal-container .xterm-viewport staat overflow-y: scroll !important — bewust toegevoegd in PR #38 om een ánder probleem te fixen: de scrollbar die verdween bij een Claude-statuswissel. Die regel forceert de native browserscrollbar (op Windows breed en dekkend) en die wordt nergens gestyled. In de xterm-DOM ligt .xterm-screen (het WebGL-/tekstcanvas) op volle breedte vóór .xterm-viewport, waar de scrollbar getekend wordt. Het canvas dekt daardoor de rechterrand van de scrollbar af — precies wat de screenshot toont. Omdat Claude Code een TUI is die zelf zijn weergave beheert, voegt een zichtbare scrollbar weinig toe; muiswiel/trackpad scrollt sowieso. Het probleem speelt op de chat-terminal; de fix moet daar gescoped blijven en de sidecar niet raken.

Planning Contract

Root cause & aanpak

De overlap is een laag-/z-order-kwestie binnen xterm: de scrollbar hoort bij de viewport-laag eronder, het tekstcanvas ligt op volle breedte erbovenop. De scrollbar verbergen verwijdert tegelijk de goot, waardoor het canvas niet meer concurreert met de scrollbar en de TUI de volle breedte krijgt. Omdat chat en sidecar dezelfde class delen, wordt het verbergen gescoped op een eigen chat-terminal-class zodat de sidecar buiten schot blijft.

Key Technical Decisions

KTD1.Verbergen via CSS, niet door scrollen uit te zetten

Voeg scrollbar-width: none toe en een ::-webkit-scrollbar { display: none }-regel. Dat verwijdert alleen de visuele track — xterm's overflow-scroll blijft actief, dus wiel/trackpad scrollt nog. Identiek aan de bestaande .scroll-hidden-utility in src/index.css (regels 162–168).

KTD2.Scope op een eigen chat-terminal-class, niet op de gedeelde regel

Chat- en sidecar-terminal renderen vandaag met dezelfde className terminal-container (zie Terminal.tsx en SidecarTerminalPanel.tsx). Een wijziging op .terminal-container .xterm-viewport zou beide raken. Voeg daarom een marker-class chat-terminal toe in Terminal.tsx en scope de verberg-regel op .chat-terminal .xterm-viewport. Gekozen boven het meeliften op het bestaande data-terminal-active-attribuut: een expliciete class benoemt de intentie en breekt niet als de sidecar-markup later verandert.

KTD3.Behoud overflow-y: scroll !important — niet terug naar auto

De chat-terminal houdt de class terminal-container, dus de bestaande overflow-y: scroll !important blijft van toepassing en hoeft niet aangeraakt. De verborgen scrollbar maakt het oorspronkelijke PR #38-symptoom (flikkerend verdwijnen) sowieso irrelevant, en scroll houdt het scrollcontainer-gedrag stabiel. Bijeffect van het verbergen: zonder zichtbare scrollbar vervalt de goot-reservering, dus de WebGL-.xterm-screen concurreert niet meer met de scrollbar en de TUI herfit naar de volle breedte — dat lost de overlap én de rechterrand-afkapping op.

Voor

// Terminal.tsx className="terminal-container absolute inset-0 bg-sidebar" /* index.css */ .terminal-container .xterm-viewport { overflow-y: scroll !important; }

Na

// Terminal.tsx className="terminal-container chat-terminal absolute inset-0 bg-sidebar" /* index.css — gedeelde regel blijft ongemoeid */ .terminal-container .xterm-viewport { overflow-y: scroll !important; } /* alleen de chat-terminal verbergt de scrollbar */ .chat-terminal .xterm-viewport { scrollbar-width: none; /* Firefox */ } .chat-terminal .xterm-viewport::-webkit-scrollbar { display: none; /* Chromium/Electron */ }
Geen JS-gedragswijziging. Alleen een class-naam wordt toegevoegd in Terminal.tsx; de safeFit/syncScrollArea-logica in useXtermInstance.ts blijft ongemoeid en houdt het scrollgebied gesynchroniseerd (scroll-naar-onder, cols/rows).
Approach & Patterns

Approach: voeg chat-terminal toe aan de className van de chat-terminal-container in Terminal.tsx (de sidecar in SidecarTerminalPanel.tsx krijgt die class níét). Voeg in index.css twee regels toe die op .chat-terminal .xterm-viewport scopen — laat de bestaande gedeelde regel staan (KTD3). In Electron is ::-webkit-scrollbar { display: none } de operatieve regel; scrollbar-width: none is onschadelijk en houdt de regel volledig.

Patterns to follow: de .scroll-hidden-utility in dezelfde file (regels 162–168) gebruikt exact dit patroon — spiegel het.

Test scenarios & Verification

Test expectation: none — CSS-/markup-only visuele wijziging zonder testbare gedragslogica (alleen een class-naam erbij). Verificatie via de Verification Contract (handmatige visuele check + bestaande suite voor non-regressie).

Verification: geen zichtbare scrollbar op de chat-terminal; sidecar-terminal toont nog wél zijn scrollbar; wiel/trackpad scrollt de chat-scrollback tot boven en onder; Claude-TUI gebruikt de volle breedte zonder rechterrand-overlap/afkapping; na een tabwissel en een Claude-statuswissel (done) werkt scroll-naar-onder nog.

Verification Contract

  • npm run test — bevestig dat de bestaande Vitest-suites (TerminalPool, projectStore) groen blijven. Er verandert geen logica; dit is de non-regressie-gate per workflowregel.
  • npm run build — schone TypeScript-/Vite-build (CSS + één className, maar bevestig geen breuk).
  • Handmatig in npm run dev: open een chat met een lange Claude-conversatie → geen zichtbare scrollbar, geen overlap, volle-breedte TUI; wiel-scroll omhoog/omlaag bereikt top en bodem; open daarna een sidecar-terminal en bevestig dat die zijn scrollbar nog tóónt (R4); trigger een statuswissel (Claude → done) en een tabwissel en bevestig dat scroll-naar-onder blijft werken (R5).
