Markdown raw/preview: scroll-positie én onopgeslagen edits overleven de wissel

Markdown wisselt vandaag van pane via een ternary die de inactieve editor unmount — scroll én edits weg. We spiegelen het bewezen HtmlEditor-patroon: één container bezit content/dirty/save/watcher (de single-source-of-truth), beide panes blijven gemount en togglen via visibility:hidden, en de content reconcilieert bij de wissel. Geen aanpak voor de HTML-editor of cross-tab scroll.

Problem Frame

De markdown-branch in src/components/Editor/EditorContainer.tsx rendert óf MarkdownEditor (Milkdown) óf CodeEditor (Monaco) via een ternary. Bij elke wissel unmount React de inactieve editor; de nieuwe mount leest het bestand vers van disk (MarkdownEditor.tsx en CodeEditor.tsx laden in een effect op mount). Eén oorzaak — de unmount gooit in-memory state weg — geeft twee symptomen: de scroll springt naar boven, en onopgeslagen edits in de verlaten pane verdwijnen. De HTML-branch loste het mount-deel al op (beide panes gemount, CSS-toggle), maar daar is de preview een read-only iframe; markdown heeft twee editable panes en vereist dus inhoudelijke reconciliatie.

Key Technical Decisions

De load-bearing keuzes. KTD1/KTD6 komen uit de brainstorm; KTD2/KTD4/KTD5 zijn afgedwongen door bestaande learnings en repo-onderzoek.

KTD1 Container bezit de state (spiegel HtmlEditor)

MarkdownEditor wordt de enige eigenaar van content, saved-content, dirty, save en file-watcher voor een markdown-tab; de twee panes worden views. Dit is de single source of truth die R5 oplost — geen twee buffers die vechten.

origin: KD1 · pattern: src/components/Editor/HtmlEditor.tsx
KTD2 Toggle via visibility:hidden, niet display:none

Beide panes blijven absoluut gepositioneerd in een relative wrapper; de inactieve krijgt visibility:hidden. display:none nuleert layout en geeft stale scroll-geometry bij reveal — wat precies het doel ondermijnt. Met visibility blijft de pane gelayout, dus scrollTop overleeft zonder JS.

learning: docs/solutions/ui-bugs/chat-scrollbar-disappears-fitaddon-fit-noop.md
KTD3 Reconciliatie bij activatie, bidirectioneel — niet continu

Milkdown is een ProseMirror-WYSIWYG met eigen state; content elke toetsaanslag als controlled prop terugvoeren reset de cursor. De panes zijn nooit tegelijk zichtbaar, dus: push de canonieke content alléén in de net-geactiveerde pane, en lees edits uit de actieve pane op.

origin: open vraag "reconciliatie-timing" → opgelost
KTD4 Vervang __milkdownReplace window-global door een per-instance handle

De huidige window.__milkdownReplace is één global die botst zodra meerdere markdown-tabs gemount zijn (last-mount-wins) — een latent bug vandaag. De reconciliatie heeft een betrouwbare handle nodig, dus we vervangen de global door een ref/imperative handle per editor-instance.

research: MarkdownEditor.tsx:154 + :231
KTD5 Eén save/keydown-handler op containerniveau, gegate op echte isActive

Met beide panes gemount zouden per-pane globale listeners dubbel afvuren (de PR #29-regressie). Eén handler in de container, gegate op de echte tab-isActive, voorkomt dat per constructie — verboden hidden panes te laten saven.

learning: docs/solutions/logic-errors/editor-save-handler-double-fire-and-isactive-propagation.md
KTD6 Round-trip fidelity voorlopig geaccepteerd

Preview→raw serialiseert via Milkdown's getMarkdown(), wat ruwe opmaak kan normaliseren (witruimte, list-markers). Niet hier opgelost; gedragen als bekend risico met een herzieningstrigger (zie Open Questions).

origin: KD4

High-Level Technical Design

Eigendoms- en datastroom-model na de rework. De container is de bron van waarheid; panes zijn views die bij activatie gesynct worden.

Reconcile-on-toggle, in één regel: als mode wijzigt, vergelijkt de container z'n canonieke content met de content van de net-zichtbaar wordende pane; verschillen ze, dan duwt de container de canonieke content in die pane (Monaco setValue / Milkdown replaceAll). Terwijl een pane actief is, schrijft z'n onChange terug naar de canonieke content. Scroll hoeft niet bewaard te worden — de pane blijft gelayout (KTD2), dus scrollTop staat er nog.

Requirements

Zeven requirements uit de origin doorgetrokken. Elke unit hieronder citeert de R-IDs die hij vervult.

Scroll-geheugen
R1Bij wisselen raw↔preview herstelt elke pane de scroll-positie van toen hij voor het laatst zichtbaar was.
R2Terugkeren naar een pane landt op de eerder bekeken positie zonder handmatig scrollen.
Edit-behoud
R3Onopgeslagen edits in de ene pane gaan niet verloren bij wisselen naar de andere en terug.
R4Edits in raw verschijnen in preview na de wissel, en omgekeerd — de actieve pane toont de meest recente content.
R5Dirty-state is consistent over beide panes: één bron per tab; Ctrl+S bewaart de meest recente content ongeacht welke pane actief is.
Geen regressies
R6Externe bestandswijzigingen (file watcher) werken beide panes consistent bij; de "overschrijf geen onopgeslagen edits"-bescherming blijft intact.
R7De eerste wissel naar preview landt op preview's eigen beginpositie (bovenaan) — cross-pane positionering wordt expliciet niet geleverd en is geen bug.

Implementation Units

Drie units. U1 en U3 zijn co-dependent (de container is dode code tot U3 hem rendert); U2 is de gedrags-laag erbovenop. Manuele verificatie staat apart hieronder.

U1

MarkdownEditor wordt een dual-pane state-owner

deps: none
Goal
Maak MarkdownEditor de enige eigenaar van content/dirty/save/file-watcher voor een markdown-tab, met een Monaco raw-view én een Milkdown preview-view tegelijk gemount, getoggeld door een mode-prop. Spiegel HtmlEditor. Dit zet R5 neer en maakt R1/R3 mogelijk.
Reqs
R1 · R3 · R5 · R6
Files
modifysrc/components/Editor/MarkdownEditor.tsx createsrc/components/Editor/MarkdownPreviewPane.tsx modifysrc/index.css testtest/MarkdownEditor.test.tsx (save-gating)
Approach
  • Spiegel HtmlEditor's lifecycle: api.fs.readFile, fileWatcherEvents subscribe, savedContentRef/currentContentRef, dirty in projectStore (setEditorDirty), en de self-write-echo-onderdrukking (pendingSelfWriteAtRef) zodat een watcher-reload geen eigen save terugdraait (R6).
  • Eén editor-save-request + Ctrl+S keydown-handler op containerniveau, gegate op de echte isActive (KTD5). Schrijft de canonieke content, ongeacht zichtbare pane.
  • Render-shell: relative wrapper; elke pane absolute inset-0; inactieve pane visibility:hidden (KTD2). Géén display:none op de panes.
  • Raw-pane: inline @monaco-editor/react <Editor language="markdown"> met dezelfde options als CodeEditor (geen herbruik van CodeEditor — die bezit z'n eigen state).
  • Preview-pane: nieuwe MarkdownPreviewPane bevat MilkdownProvider/Milkdown + de huidige plugins (commonmark, gfm, history, clipboard, indent, cursor, listener, taskCheckboxToggle) en exposeert een replace(content) handle via ref — vervangt window.__milkdownReplace (KTD4).
Patterns
src/components/Editor/HtmlEditor.tsx (state-owner, watcher, echo-suppress, dirty, beide panes gemount) · huidige Milkdown-setup in MarkdownEditor.tsx · Monaco-options in CodeEditor.tsx.
Test scenarios
  • HappyContainer met isActive=true: editor-save-request event → api.fs.writeFile aangeroepen met de canonieke content.
  • ErrorContainer met isActive=false: editor-save-request + Ctrl+S keydown → writeFile NIET aangeroepen (hidden/inactive tab saved niet — de PR #29-regressie).
  • IntegWatcher file-changed met schone buffer → content state ververst; met dirty buffer én afwijkende disk → edits behouden + console-warn (mirror HtmlEditor).
Verification
Markdown-tab opent met beide panes in de DOM (één visibility:hidden); Ctrl+S in raw bewaart; watcher-reload draait een eigen save niet terug. Scroll-/layout-gedrag wordt manueel geverifieerd (zie Manual Verification) — niet headless-testbaar.
U2

Bidirectionele content-reconciliatie bij de toggle

deps: U1
Goal
Zorg dat edits over de panes heen meekomen: bij mode-wissel duwt de container de canonieke content in de net-actieve pane (alleen bij verschil); de actieve pane schrijft via onChange terug. Dit levert R3/R4 en bewaakt de single dirty-source (R5).
Reqs
R3 · R4 · R5
Files
createsrc/utils/editorReconcile.ts (pure helpers) modifysrc/components/Editor/MarkdownEditor.tsx testtest/editorReconcile.test.ts
Approach
  • Extraheer de beslislogica naar pure functies in editorReconcile.ts zodat ze headless testbaar zijn (CLAUDE.md: schrijf geen code die je niet kunt testen): bv. needsSync(activePaneContent, canonical) en computeDirty(current, saved).
  • useEffect op mode: bij activatie van een pane, als needsSync → push canonical via de pane-handle (Monaco setValue + cursor-herstel zoals de bestaande watcher-reload; Milkdown replace). Inactieve pane blijft bevroren.
  • Actieve pane's onChange updatet currentContentRef + dirty. Bij de wissel wordt eerst de uittredende pane's laatste content uitgelezen en canonical gemaakt, dán de intredende pane gesynct.
  • Let op de reconcile-volgorde-discipline (restore ná subscribe) uit de terminal-LRU-learning: lees-uittredend → schrijf-intredend, niet andersom.

Test scenarios
  • HappyneedsSync('a', 'b') → true; needsSync('a', 'a') → false (geen onnodige replace die de cursor reset).
  • HappyCovers AE2. Canonical bevat een nieuw geschreven heading → na sync bevat de doel-pane die heading.
  • EdgecomputeDirty(current, saved): gelijk → false; afwijkend → true; lege string vs lege string → false.

Manual Verification

Scroll- en layout-gedrag is niet headless-unit-testbaar (zelfde klasse als de FitAddon-learning). Verplichte handmatige pass vóór merge:

  • Scroll-behoud (R1/R2)Scroll raw één scherm omlaag → Preview → Raw: zelfde positie, geen sprong.
  • Edit-behoud (R3/R4)Onopgeslagen edits in raw overleven een wissel naar preview en terug, en verschijnen in preview.
  • Eerste peek (R7)Vers geopend doc → Preview staat bovenaan; bevestig dat dit als verwacht voelt.
  • File-watcher (R6/AE5)Laat Claude het bestand wijzigen met schone buffer → beide panes tonen nieuwe content; met dirty buffer blijven edits staan.
  • Multi-tabTwee markdown-tabs open: reconciliatie/dirty/save gaan naar de juiste tab (verifieert dat KTD4 de global-collision oplost).
  • Split-view & layoutGeen layout-regressie in split-view of bij tab-switch; Monaco rendert correct na reveal (geen 0-breedte).

Scope Boundaries

Uitgesteld (later mogelijk)

  • Cross-pane scroll-syncLanden op exact dezelfde inhoudelijke plek in de andere view (proportioneel of regel-gemapt).
  • Cross-tab scroll-geheugenScroll bewaren wanneer je een hele editor-tab verlaat en terugkomt (tab-niveau gebruikt nog display:none).
  • Round-trip fidelity oplossenRaw als canoniek behandelen zodat Milkdown opmaak niet normaliseert (KTD6).

Buiten deze feature

  • HTML-editorPreserveert per-pane scroll al; read-only preview, geen edit-verlies. Geen wijziging.
  • Brede display:none→visibility refactorAlleen de markdown raw/preview-toggle krijgt visibility; andere editors/tab-niveau blijven ongemoeid.
  • Nieuwe hotkeyGeen nieuwe gebruikersactie geïntroduceerd; de bestaande toggle verbetert alleen.
