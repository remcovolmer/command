feat: Pinnable projects vervangen het workspace-type

Datum: 2026-06-11 Type: feat / refactor Depth: Standard Origin: docs/brainstorms/2026-06-10-pinnable-projects-requirements.html Status: completed

Samenvatting

Splits "bovenaan staan" los van "feature-set": pinned wordt een vlag op elk project, het workspace-type verdwijnt.

Zes dependency-geordende units. Eerst additieve plumbing (pin-vlag, IPC, store-actie, migratie), dan de sidebar-render en de pin-affordances, en als laatste het verwijderen van het workspace-type-lid zodra geen enkele referentie er nog naar wijst. De aanpak leunt op drie bestaande patronen — het project:reorder-IPC-kanaal, de versie-voor-versie migratiechain in ProjectPersistence, en de drie-bestand hotkey-registratie — plus twee institutionele learnings die de migratie en de sidebar-refactor sturen.

Probleemkader

Het type doet vandaag twee dingen tegelijk; we knippen die twee assen los.

Situatie

ProjectType = 'workspace' | 'project' | 'code'. Het workspace-type wordt apart gerenderd: eigen blok bovenaan, ster, altijd-zichtbare chats (~80 regels in Sidebar.tsx).

Complicatie

Positie (bovenaan) zit vast aan feature-set. Een code-project bovenaan pinnen kan niet zonder worktrees/git te verliezen. Vier verspreide type === 'workspace'-checks dragen verborgen gedrag.

Resolutie

Een orthogonale pinned: boolean-vlag. Positie en feature-set worden onafhankelijk; gepinde projecten renderen via het bestaande SortableProjectItem; het type-lid workspace wordt verwijderd.

Requirements (traceability naar origin)

  1. Pin is orthogonaal aan type — zowel project als code zijn pinbaar; een gepind code-project behoudt worktrees, git-tab, sidecar en GitHub-menu. → U1, U3
  2. Eigen "Pinned"-sectie bovenaan, alleen zichtbaar bij ≥1 pin. → U3
  3. Gepinde projecten renderen via het bestaande SortableProjectItem — geen apart codepad. → U3
  4. Drie (de)pin-paden: hover-ster, context-menu "Pin to top"/"Unpin", en hotkey Ctrl+Shift+P voor het actieve project. → U4
  5. Volgorde binnen de Pinned-sectie is drag-herordenbaar, via dezelfde sorteer-infra als de bestaande lijst. → U3
  6. Pin-status persisteert over herstart (op disk in projects.json). → U1
  7. Eenmalige migratie: bestaand type:'workspace' → type:'project' + pinned:true, zonder verlies van naam/pad/settings. → U2
Bevestigde gedragsveranderingen uit de brainstorm (carry-forward): (1) always-expanded terminals vervalt — gepinde projecten gedragen zich als normale projecten; (2) het special-case workspace-renderblok wordt verwijderd, niet behouden. Beide expliciet akkoord (see origin).

Key Technical Decisions

KTD1 · Pin = pinned: boolean, volgorde via bestaande sortOrder

Waarom: minimale schemawijziging. Within-section volgorde hergebruikt reorderProjects → project:reorder (sortOrder-gebaseerd), exact zoals de active/inactive-secties. Geen apart pinnedAt-veld nodig. (open question uit origin — hierbij beslist)

KTD2 · Dedicated project:setPinned IPC-kanaal (niet project:update)

Waarom: project:update + ProjectPersistence.updateProject whitelisten alleen name/settings en negeren stilletjes onbekende velden — een pin-toggle zou no-oppen. Een eigen kanaal spiegelt het bestaande project:reorder-patroon: main-handler + preload + ProjectPersistence.setProjectPinned + store-actie.

KTD3 · Twee-oppervlak idempotente migratie; ProjectPersistence is source of truth

Waarom: de schijf (projects.json, STATE_VERSION 6→7) is de waarheid; onRehydrateStorage in de store is een defensieve fallback voor een persisted snapshot. Eén oppervlak migreren laat de ander stale (learning: file-explorer-worktree-awareness.md). Beide transforms zijn idempotent: type==='workspace' → transform, anders no-op.

KTD4 · Pin-toggle + side-effects in één store-actie; UI-paden zijn dunne callers

Waarom: ster, context-menu én hotkey roepen dezelfde projectStore-actie aan. Divergente logica per invocatiepad is een gedocumenteerde bug (learning: hotkey-handler-missing-auto-switch-behavior.md). De actie bevat ook de fallback-selectie als het actieve project uit beeld raakt.

KTD5 · Pinned rendert via SortableProjectItem (derde DndContext-sectie), geen parallel codepad

Waarom: de Pinned-sectie spiegelt de Active/Inactive-secties in SortableProjectList (eigen DndContext + createDragEndHandler). Een eigen render met eigen empty-state liet de twee paden eerder uit elkaar lopen (learning: workspace-empty-state-visibility.md). Bewaar de guard isActive && terminals.length === 0.

Implementation Units (dependency-geordend)

U1

Pin-vlag + IPC-kanaal + persistence + store-actie (additief)

GoalVoeg de pinned-vlag en de volledige mutatie-plumbing toe zonder bestaand gedrag te breken. 'workspace' blijft voorlopig in de union.

RequirementsR1, R6 · KTD1, KTD2, KTD4

Dependencies— (start)

Filessrc/types/index.ts electron/preload/index.ts electron/main/index.ts electron/main/services/ProjectPersistence.ts src/stores/projectStore.ts

ApproachVoeg pinned?: boolean toe aan de Project-interface (alle 3 type-declaraties + canonical). Nieuw project:setPinned(id, pinned) handler in electron/main/index.ts met UUID-validatie, gespiegeld in preload (project-namespace) en de electronAPI-interface. ProjectPersistence.setProjectPinned(id, pinned) zet de vlag en roept saveState(). addProject default pinned: false. Store-actie setPinned/togglePin in de async stijl van reorderProjects: roep IPC, re-fetch list(), set({projects}). De actie bevat ook de fallback-selectie (zie U5-logica) — maar die logica leeft hier al in de actie, niet in callers.

Patternsproject:reorder handler + reorderProjects store-actie als template voor het hele kanaal.

TEST SCENARIOS
  • Happy: setPinned(id, true) op een ongepind project → IPC aangeroepen met (id, true), store re-fetcht, project heeft pinned:true.
  • Toggle: tweemaal togglen → terug naar uitgangsstaat.
  • Edge: setPinned op onbekende id → geen crash, geen mutatie (handler valideert id).
  • Persistence: setProjectPinned schrijft de vlag en bewaart; herladen geeft pinned terug.

Pin-vlag is end-to-end zetbaar via de store-actie en overleeft een herstart; nog geen UI.

U2

Eenmalige workspace→pinned migratie (disk + defensieve store-guard)

GoalBestaande type:'workspace'-projecten worden idempotent type:'project' + pinned:true — op schijf én in een persisted store-snapshot.

RequirementsR7 · KTD3

DependenciesU1 (pinned-veld bestaat)

Fileselectron/main/services/ProjectPersistence.ts src/stores/projectStore.ts

ApproachBump STATE_VERSION 6→7. Voeg een version===6-stap toe aan de migrateState-chain die elk project mapt: p.type==='workspace' ? {...p, type:'project', pinned:true} : p. Werk de default-fallthrough-tak bij zodat die het pinned-veld meeneemt. In projectStore onRehydrateStorage: defensieve guard die dezelfde transform toepast op een persisted snapshot (no-op als al gemigreerd). ProjectPersistence is de bron; de store-guard is vangnet.

PatternsDe bestaande v2→3-stap (mapt projecten naar type:'code' + defaults) als template. onRehydrateStorage expandedPaths-migratie als template voor de store-guard.

TEST SCENARIOS
  • Covers R7. Migratie: v6-state met een type:'workspace'-project → na migratie type:'project', pinned:true, naam/pad/settings ongewijzigd, version===7.
  • Idempotent: al-gemigreerde state (geen workspace, pinned gezet) → transform laat alles ongemoeid; tweede run = no-op.
  • Niet-workspace: code/project zonder pin → blijft ongepind, type ongewijzigd.
  • Store-guard: persisted snapshot met workspace-vorm → na rehydrate genormaliseerd, identiek aan disk-resultaat.

Een bestaande workspace verschijnt na opstarten als gepind project; herhaald opstarten verandert niets.

Risico's & mitigatie

Migratie-dataverlies / niet-idempotent. Mitigatie: idempotente transform met expliciete type==='workspace'-guard; tests in beide richtingen (U6); atomic saveState (temp+rename) bestaat al.
Stale persisted store-cache (twee oppervlakken). Mitigatie: onRehydrateStorage-guard naast de disk-migratie (KTD3); store-guard-test (U2).
Gemiste 'workspace'-referentie → "geselecteerd maar onzichtbaar"-bug. Mitigatie: volledige enumeratie (8 sites) + grep-sweep met nul-treffer-gate in U5; gesloten ProjectType-union dwingt compile-fouten af bij gemiste refs.
Hotkey vuurt niet in tekst-input. Status: geaccepteerd — useHotkeys bailt op INPUT/TEXTAREA/contentEditable; voor een pin-actie prima (learning #5).

Naar implementatie doorgeschoven

  • Exacte helper-/methodenamen voor de migratiestap en de store-actie (setPinned vs togglePin).
  • Of de fallback-selectie ook een actie-side-effect nodig heeft wanneer je het actieve project unpinned (waarschijnlijk niet — unpinnen verbergt niet, het verplaatst alleen) — bevestigen tijdens implementatie tegen de echte render.
  • Definitieve plek van de nieuwe IPC-mock-shape in de testharness.

Geen externe research uitgevoerd: sterke lokale patronen voor elk raakvlak (IPC-kanaal, migratiechain, hotkey-registratie), geen externe contracten geraakt.
