Product Contract

Summary

Command krijgt één echte browser op basis van een Electron <webview>-tag als eersteklas center-tab. HTML-bestanden openen daar in plaats van in de sandboxed editor-preview, en de bestaande Ctrl+Shift+B-browser wórdt diezelfde ene surface. De browser ververst automatisch bij een wijziging op schijf en biedt echte browser-capabilities — storage, cookies, same-origin fetch, dev tools en navigatie. Het oude srcDoc-preview-pad verdwijnt.

Problem Frame

De huidige preview is een bewust dichtgetimmerd srcDoc-iframe (HtmlPreview.tsx): null-origin, geen allow-same-origin, een herschreven CSP, geen navigatie en geen dev tools. Veilig, maar functioneel arm — pagina’s die localStorage, cookies, same-origin fetch, service workers of dev tools nodig hebben werken er simpelweg niet.

Daarnaast bestaan er twee losse HTML-render-paden die hetzelfde probleem half oplossen: de editor-preview én een apart iframe-gebaseerd BrowserTab. Elke feature rond de browser — en de agent die ’m straks moet kunnen aansturen — zou daardoor twee keer gebouwd en onderhouden moeten worden. De kosten zijn dubbel onderhoud nu, en een geblokkeerde ambitie later.

Key Decisions

KD1Engine: Electron <webview>-tag, niet WebContentsView

De <webview> leeft in de DOM zoals het huidige iframe, dus tabs, split-view, dialogs en flyouts blijven werken met minimale layout-verandering. Het is een echt browserproces (storage, cookies, dev tools, navigatie) en geeft main-process toegang voor latere agent-haken via getWebContentsId() + webContents.fromId(). WebContentsView geeft betere main-process controle, maar is een native overlay die DOM-stacking negeert — forse, doorlopende plumbing om ’m onder tabs, panelen en dialogs te positioneren. Herzien alleen als agent-automation dat later afdwingt.

KD2Browser = eersteklas tab; bekijken ≠ bewerken

Dubbelklik op een .html-bestand opent de browser-tab. De HTML-bron bewerken is een expliciete, aparte actie (“Open als code” → Monaco raw-editor). Schoon mentaal model, en het beste fundament voor agent-features: de browser is niet langer ondergeschikt aan de editor.

KD3Één geünificeerde browser

De echte <webview> vervangt zówel de srcDoc-preview als het iframe in BrowserTab. Alle ingangen funnelen naar dezelfde surface. “Niet twee keer doen”: één render-pad om te onderhouden en om agent-features op te bouwen.

KD4Externe URL’s laden wél, maar worden niet gehard of ondersteund

De adresbalk laadt elke URL die je typt — file://, localhost én externe http(s). Een echte webview maakt dat sowieso mogelijk (de X-Frame-Options-muur van het iframe valt weg), dus externe pagina’s bewust blókkeren zou alleen nodeloze frictie en afdwing-werk opleveren. Maar we ondersteunen ze ook niet: geen browser-grade veiligheid, chrome of persistentie (history, downloads, permissie-UI). Unpolished, op eigen risico. Command blijft van identiteit een lokale-HTML + localhost-viewer; algemeen internetten is bijvangst, geen doel. Omdat er nu ook onvertrouwde content kan laden, weegt de isolatie-eis (R12) juist zwaarder.

Key Flows

F1 — HTML openen. Trigger: gebruiker dubbelklikt een .html/.htm-bestand (file explorer), klikt een OSC 8 .html-link of Ctrl+klikt een HTML-pad in een Claude-chat, of dropt een HTML-bestand vanuit de sidebar. De browser-tab opent (of wordt geactiveerd) en rendert het bestand via de webview met echte browser-capabilities.

F2 — Live-reload. Trigger: het bestand dat in de browser geladen is verandert op schijf (Claude hergenereert een deliverable, of de gebruiker slaat de bron op). De FileWatcherService meldt de wijziging; de browser-tab herlaadt automatisch.

F3 — Bron bewerken. Trigger: gebruiker kiest “Open als code” op een HTML-bestand. De Monaco raw-editor opent in een editor-tab. Na opslaan herlaadt een reeds-open browser-tab van hetzelfde bestand (via F2).

Requirements

Browser-surface & engine

  • R1Er is één browser-surface, gerenderd door een Electron <webview>, beschikbaar als eersteklas center-tab — en in het tweede paneel, zoals de huidige BrowserTab.
  • R2De webview draait een echt browserproces: localStorage, cookies, same-origin fetch, service workers en dev tools werken zoals in Chrome.
  • R3De bestaande srcDoc-preview (HtmlPreview) en het iframe in BrowserTab worden vervangen door deze ene webview; er blijft geen tweede HTML-render-pad bestaan.

Routing & ingangen

  • R4Een .html/.htm-bestand openen opent de browser-tab, niet langer een editor-tab met preview. Geldt voor dubbelklik in de file explorer, OSC 8-links én Ctrl+klik-paden op .html/.htm in een Claude-chat, en drag-drop vanuit de sidebar.
  • R5.md-links en niet-HTML-bestanden blijven routeren zoals nu (markdown-editor resp. Monaco). Alleen HTML verandert van bestemming.

Live-reload

  • R6Wanneer het bestand dat in de browser geladen is op schijf verandert, herlaadt de browser automatisch. Hergebruikt de bestaande FileWatcherService (fs:watch:changes).
  • R7Live-reload geldt ongeacht de bron van de wijziging — Claude die een deliverable hergenereert, of de gebruiker die de bron opslaat via “Open als code”.

Bewerken van de bron

  • R8Er is een expliciete actie om een HTML-bestand als bewerkbare broncode te openen (Monaco raw-editor). Default-ingang: rechtsklik in de file explorer → “Open als code”.
  • R9De bron bewerken en opslaan laat een reeds-open browser-tab van hetzelfde bestand live herladen (via R6).

Navigatie-chrome

  • R10De browser-tab heeft werkende nav-chrome tegen de echte webview: terug, vooruit, herladen, en een adresbalk. De adresbalk laadt file://-paden, http(s)://localhost én externe http(s)-URL’s — externe pagina’s laden op eigen risico en worden niet gehard (zie KD4).
  • R11Dev tools zijn te openen op de webview-inhoud.

Beveiliging

  • R12De geladen pagina mag de Electron preload-bridge of de app-shell niet kunnen bereiken (geen Node-integratie, geïsoleerd) — behoud van de bewuste bridge-escape-verdediging die de huidige sandbox biedt. Weegt zwaarder nu de webview ook externe, onvertrouwde pagina’s kan laden (KD4).

Acceptance Examples

AE1  Covers R4, R6, R7.

Gebruiker dubbelklikt docs/plans/…-plan.html → het opent in de browser-tab en rendert de CDN-charts en webfonts zoals een echte browser. Claude hergenereert hetzelfde bestand → de tab herlaadt automatisch.

AE2  Covers R8, R9.

Gebruiker rechtsklikt een .html → “Open als code” → de Monaco raw-editor opent in een editor-tab. Bewerkt en Ctrl+S → de open browser-tab van datzelfde bestand herlaadt.

AE3  Covers R2.

Een HTML-pagina die met localStorage.setItem een waarde bewaart en die bij herladen weer uitleest, behoudt de waarde binnen dezelfde tab-sessie — wat in de oude null-origin sandbox niet werkte.

AE4  Covers R10.

In de adresbalk localhost:5173 typen laadt de lokale dev-app; file://…/rapport.html laadt een lokaal bestand.

Success Criteria

  • Een gegenereerde HTML-deliverable (CDN-charts, webfonts, inline scripts) rendert visueel identiek aan openen in Chrome.
  • localStorage, cookies, same-origin fetch en service workers functioneren binnen de webview.
  • Dev tools openen op de webview-inhoud.
  • Hergenereren of opslaan van het bestand ververst de zichtbare browser-tab zonder handmatige actie.
  • Geen enkele geladen pagina kan window.electronAPI of de app-shell bereiken (bridge-escape-test blijft groen).

Scope Boundaries

Uitgesteld (fundament sluit het niet uit)
  • Agent↔browser-integratie — pagina-DOM/screenshot naar Claude, agent die zelf navigeert. Indien gebouwd: via het wrappen van de Claude Code CLI, nooit SDK/API (terminal-first beslissing, 2026-07-02).
  • Migratie naar WebContentsView — alleen herzien als agent-automation zwaardere main-process controle eist.
  • Browser-grade ondersteuning voor externe pagina’s — navigatie-/popup-/download-handling, permissie-handlers, certificaat- en mixed-content-beleid, history/persistentie. Externe URL’s laden wél (KD4), maar worden niet gehard; dít is de werklijst als we er ooit een echte browser van willen maken.
Buiten de identiteit van deze iteratie
  • Een volwaardige, geharde webbrowser worden (met bijbehorende veiligheids- en beheer-UI). Externe URL’s laden op eigen risico, maar we bouwen er geen browser omheen — dit blijft een lokale-HTML + localhost-viewer.

Planning Contract

Key Technical Decisions

KTD1Engine <webview> (zie Product Contract KD1)

Bevestigd in de brainstorm. DOM-embedded, minimale layout-impact, echt proces, main-process toegang voor latere agent-haken. Deze KTD's werken KD1 uit tot een implementeerbaar ontwerp.

KTD2Guests hardenen in het main-proces, niet via <webview>-attributen

De beveiliging (R12) leeft in het main-proces, niet in de renderer-markup. Zet webviewTag: true op het renderer-venster (nu afwezig, electron/main/index.ts:316) en vang elke guest op via app.on('web-contents-created') → will-attach-webview: forceer nodeIntegration:false, contextIsolation:true, strip elke preload, en pin de partition. Op de guest-webContents: setWindowOpenHandler (popups weren/afvangen) en een will-navigate-beleid. Rationale: element-attributen worden door renderer-markup gezet en zijn geen betrouwbare grens voor onvertrouwde content; main-process-interceptie is de afdwingbare chokepoint.

KTD3Dedicated persistente partition, geïsoleerd van de app

De webview draait op een eigen persistente partition (bijv. persist:command-browser), gescheiden van de sessie van de Electron-app. Cookies en localStorage overleven restarts (echt browsergedrag, dient R2/AE3) terwijl de app-storage ongemoeid blijft. Bevestigd boven ephemeral in-memory.

KTD4Één gedeelde pad→file://-helper

Extraheer de Windows-bewuste normalisatie van buildBaseHref uit HtmlPreview.tsx naar een util (bijv. src/utils/fileUrl.ts) vóórdat het component in U5 verdwijnt, zodat routing en toekomstige consumers één implementatie delen. Voorkomt dat de drive-letter-met-drie-slashes-bug opnieuw ontstaat.

KTD5Extensie-split routing met de OSC 8-classifier als enige security-chokepoint

Voeg een browser-beslissing toe aan osc8LinkRouter.ts voor .html/.htm (behoud editor voor .md), i.p.v. in de activate-handler op rauwe strings te branchen — de traversal-/scheme-/lengte-checks blijven op één geteste plek. File explorer en sidebar branchen op extensie bij hun eigen open-call-sites.

KTD6Live-reload hergebruikt de bestaande file-watch-stream

Abonneer op de bestaande per-project fs:watch:changes-events (FileWatcherService watcht al de projectmap), gefilterd op het geladen bestandspad en gedebounced (~200ms). Géén tweede watcher. Rationale: een per-bestand-watcher zou dupliceren en de gedocumenteerde chokidar-valkuilen riskeren (startup-leak, v4 glob-ignore-breuk — zie Sources).

High-Level Technical Design

Sequencing

U1 → U2 → U3 → { U4, U5, U6 }. Eerst de webview veilig aanzetten (U1) en het component omzetten (U2); dan de routing + gedeelde helper (U3); daarna parallel: chat-routing (U4), het oude pad opruimen (U5), en live-reload (U6).

Assumptions

  • webviewTag staat standaard uit en moet op het renderer-venster aan (bevestigd afwezig op electron/main/index.ts:316).
  • De bestaande per-project fs:watch:changes-stream volstaat voor live-reload; geen per-bestand-watcher nodig.
  • De sandbox:false op het hoofdvenster (nodig voor node-pty) verzwakt de guest-isolatie niet — guests dragen hun eigen webPreferences met Node uit (te verifiëren in U1).

Open Questions (deferred to implementation)

  • Split-view met twee gelijktijdige <webview>-guests — verifiëer dat beide zich correct gedragen (elk eigen webContents); verwacht goed.
  • Exacte component/plek van de “Open als code”-context-menu-actie (gerenderd door de parent van FileTreeNode via de onContextMenu-callback).
  • Live-reload: volledige reload() vs. behoud van scrollpositie/dev-tools-state — default volledige reload voor v1.

Risks, Dependencies & System-Wide Impact

Risk · accepted

<webview> wordt door Electron officieel ontmoedigd (perf, quirks, apart proces). Mitigatie: gebruik geïsoleerd achter BrowserTab; herzien WebContentsView alleen als het knelt (Scope Boundaries).

Risk · mitigated (isolation) / deferred (hardening)

Externe, onvertrouwde content vergroot het aanvalsoppervlak. Mitigatie: main-process guest-hardening (U1) + bridge-escape-test (R12). Externe pagina’s blijven bewust ongehard (KD4).

Risk · mitigated

Hergebruik van de file-watch-stream erft chokidar-valkuilen (v4 glob-ignore-breuk, startup-leak — zie Sources). Mitigatie: abonneer alleen op bestaande events, voeg geen watcher toe, debounce.

System-wide impact

webviewTag aanzetten verbreedt het capability-oppervlak van de renderer (elke <webview> in de app). Het content-tab-systeem, de chat-link-afhandeling en de file explorer veranderen alle drie het open-gedrag voor HTML. Begrensd, maar cross-cutting.

Verification Contract

  • Unit tests: npm test (Vitest) — osc8LinkRouter, BrowserTab, store-routing, live-reload-matcher. Gericht: npm test -- <path>.
  • Typecheck/build: npm run build (tsc + vite + electron-builder) moet groen zijn; geen any (ESLint).
  • Smoke (dev app / Playwright-Electron): open een gegenereerde HTML-deliverable (charts/fonts renderen als Chrome); dev tools openen; “Open als code” + opslaan → tab herlaadt; Claude hergenereert het bestand → tab herlaadt; externe URL typen → laadt; bridge-escape-check (guest kan electronAPI niet bereiken).
  • Native modules: npm run rebuild alleen indien node-pty geraakt wordt (niet verwacht).

Definition of Done

  • Alle unit-test-scenario’s slagen; npm test + npm run build groen.
  • Bridge-escape geverifieerd: geen guest-pagina bereikt de preload-bridge/host.
  • Één HTML-render-pad — HtmlPreview/srcDoc verwijderd, geen resterende imports.
  • Dubbelklik / OSC 8 / Ctrl+klik / drag-drop van .html openen de browser; .md opent de editor.
  • Live-reload werkt bij hergenereren én bij opslaan vanuit de raw-editor.
  • Externe URL laadt (ongehard, per KD4); dev tools + nav-chrome werken.
  • Afgebroken/experimentele code is uit de diff verwijderd.
