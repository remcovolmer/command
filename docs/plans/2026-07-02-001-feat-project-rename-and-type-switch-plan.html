Goal Capsule

Objective
Make sidebar projects editable in place — rename the display name and switch a project between the Project and Code types — without leaving the sidebar, touching the filesystem, or restarting sessions.
Product authority
Remco — in-session dialogue, 2026-07-02.
Open blockers
None. Both prior deferred questions (F2 binding scope, type-switch hotkey) were resolved during planning — see KTD3 and KTD5.

Product Contract

Product Contract preservation: unchanged. All R-IDs, flows, and acceptance examples are carried verbatim from the requirements-only version; planning added the HOW below without altering product scope.

Summary

Add in-place project rename (inline row edit via double-click, F2, or a context-menu Rename entry) and a single context-menu toggle that switches a project between Project and Code. Both are label/config edits — nothing on disk moves and no Claude session or terminal restarts.

Problem Frame

A project's name is fixed to the folder basename when it is added, and its type — project (Files + Claude) or code (full dev tools) — is chosen once in the Add Project dialog. Neither can change afterward. A folder with an awkward or ambiguous name is stuck with that label in the sidebar, and a project added as the wrong type has to be removed and re-added to get the right panels (git tab, sidecar shell). The rename path already exists end-to-end in the backend but has no way to trigger it from the UI; type has no edit path at all.

Key Product Decisions

KD1Rename is a display label, decoupled from disk

The name lives only in Command's project record. The folder path, worktrees, and running sessions are never touched. Accepted cost: the sidebar name can diverge from the actual folder name.

KD2One-click type toggle, not a submenu

Only two types exist, so a single context-menu item that flips to the other type is simpler and faster than a submenu or radio group.

KD3Type switch is reactive — no restart

Panel visibility (git tab, sidecar shell, activity rail) is already derived from the project's type at render time, so switching updates the UI immediately without respawning terminals or Claude sessions.

KD4Inline edit mirrors the file-explorer rename

Double-click / F2 to start, Enter to commit, Esc to cancel — the same interaction the file explorer already uses for renaming files, so the pattern is familiar and consistent.

Requirements

Rename

R1

A project's name can be edited inline from the sidebar: double-clicking the name, or pressing F2 while the project is active, replaces it with a text input pre-filled with the current name.

R2

The project right-click context menu includes a Rename entry that starts the same inline edit.

R3

Enter or blur commits the edit; Esc cancels and restores the previous name. A blank or whitespace-only value is rejected and the previous name is kept.

R4

Renaming changes only the display name. The folder path, worktrees, and running sessions are unaffected, and duplicate names across projects are allowed.

Change type

R5

The project context menu includes one toggle entry that switches the project to the other type: Switch to Code when it is currently a Project, Switch to Project when it is currently Code.

R6

The switch takes effect immediately with no session or terminal restart; the sidebar icon, git tab, sidecar shell, and activity rail reflect the new type reactively.

R7

Switching Code → Project hides the git tab and sidecar shell and falls back to the files tab; switching back restores them. No confirmation or warning is shown on the downgrade.

Persistence & consistency

R8

Name and type changes persist across app restarts and stay consistent everywhere the project is shown — sidebar row, project overview, and the type icon (folder vs. code glyph).

Key Flows

F1. Rename a project · Covers R1–R4

  1. Trigger: double-click the name, press F2 on the active project, or pick Rename from the context menu.
  2. The name becomes a text input pre-filled with the current name, text selected.
  3. User edits. Enter or blur commits; a blank value is rejected and the previous name kept. Esc cancels.
  4. The sidebar row shows the new name; the folder on disk is untouched.

F2. Switch a project's type · Covers R5–R7

  1. Trigger: pick Switch to Code / Switch to Project from the context menu.
  2. The type flips on the project record and persists.
  3. The UI re-renders reactively: icon updates, and the git tab / sidecar / activity rail appear or hide per the new type.
  4. On a Code → Project downgrade, if the git tab was active the view falls back to the files tab. Running chats and terminals are unaffected.

Acceptance Examples

AE1· Covers R3

Blank name rejected. Start a rename, clear the field, press Enter → the edit closes and the previous name is retained.

AE2· Covers R3

Esc cancels. Start a rename, type a different name, press Esc → the previous name is retained, no update is persisted.

AE3· Covers R4

No disk impact. Rename a project whose folder is foo to "My App" → the folder on disk stays foo, worktrees resolve, and any running chat keeps working.

AE4· Covers R6, R7

Downgrade hides git tab. On a Code project with the git tab active, choose Switch to Project → the git tab is hidden, the files tab is shown, no warning appears. Choosing Switch to Code restores the git tab.

AE5· Covers R6

No restart on switch. Switch a project's type while a Claude chat is running in it → the chat and its PTY are unaffected.

Scope Boundaries

  • Renaming the folder on disk — and the resulting path, worktree, and session rewrite — is out of scope.
  • No new project types beyond Project and Code.
  • No bulk rename or bulk type-change across multiple projects.
  • Worktree renaming is unchanged; it is a separate concept with its own naming.

Assumptions

A1

Changing type is a config-only update to the persisted project record — no data migration and no filesystem change.

A2

Type already drives panel visibility through a reactive selector (isLimitedProject in FileExplorer.tsx; ActivityRail.tsx), so no imperative teardown/rebuild is needed on switch. Verified in code.

Planning Contract

Approach

The rename backend already exists end-to-end for the name field (project:update IPC → ProjectPersistence.updateProject → store updateProject). The work is: (1) extend that same update path to also accept type; (2) add store-level rename state mirroring the file explorer's startRename pattern so a global F2 can drive the active project's row; (3) build the inline-edit UI and the two context-menu entries in the sidebar row; (4) register the two hotkeys. No new IPC channels, no schema/persistence-shape changes, no migration.

Key Technical Decisions

KTD1Extend the existing project:update path for type

Add type to the four places the name update already flows through — ProjectPersistence.updateProject, the project:update IPC allowedUpdates whitelist (validating 'project' | 'code' only), the electronAPI.project.update signature, and the store's updateProject Pick<>. One validated channel; no new surface.

KTD2Rename state lives in the store, not local component state

Mirror the file explorer's startRename / fileExplorerRenamingPath pattern with a renamingProjectId and a startProjectRename(id) action. This lets the global F2 handler (in App.tsx) and the double-click / context-menu triggers all route through one action, and lets exactly one row be in edit mode at a time.

KTD3F2 is shared with file-explorer rename — dispatch by focus zone (resolves prior Q1)

Bind F2 to a new sidebar.renameProject action whose handler renames the active project only when the sidebar is the focused zone; the existing fileExplorer.rename handler already no-ops without a selected file. If the hotkey conflict detector rejects a duplicate F2 across categories, collapse to a single F2 handler that branches on the active focus zone. Double-click and the context-menu Rename entry remain the primary, unambiguous triggers.

KTD4Type switch needs no imperative teardown

isLimitedProject (FileExplorer) and ActivityRail derive from project.type via reactive Zustand selectors; flipping the persisted type re-renders the affected panels. FileExplorer already falls back to the files tab when the git tab is active on a limited project (activeTab={isLimitedProject && activeTab === 'git' ? 'files' : activeTab}), so R7's downgrade fallback is already handled by existing code.

KTD5Ship a type-switch hotkey to honor the repo's hotkey convention (resolves prior Q2)

The repo requires every user-facing feature to carry a keyboard shortcut. Add sidebar.toggleProjectType that flips the active project's type. Assign a non-conflicting default binding and verify it against the existing conflict detector, adjusting the key if it clashes.

Implementation Units

U1

Extend the project update path to accept type

Goal
Let type flow through the existing name-update path, persisted and validated.
Requirements
R5, R6, R8
Dependencies
None (foundation)
Files
electron/main/services/ProjectPersistence.ts, electron/main/index.ts, electron/preload/index.ts, src/types/index.ts, src/stores/projectStore.ts, test/projectPersistence.test.ts
Approach
  • In updateProject, persist a valid type when present, alongside the existing name/settings handling.
  • In the project:update IPC handler, add type to allowedUpdates only when it equals 'project' or 'code' (reject anything else — do not fall through to a default).
  • Widen the electronAPI.project.update parameter type and the store updateProject Pick<Project, 'name' | 'settings'> to include 'type'.
  • No any — use ProjectType and narrow the IPC input with an explicit value check.
Test scenarios
  • Happy path: updateProject(id, { type: 'code' }) persists the new type and returns the updated project.
  • Edge: updateProject(id, { name, type }) updates both fields in one call.
  • Error path: an invalid type value (e.g. 'xyz') is not written; the stored type is unchanged.
  • Error path: updateProject on an unknown id returns null (existing behavior preserved).
Verification

projectPersistence.test.ts passes; a type update survives a persistence reload; typecheck clean on the widened signatures.

Add rename state + startProjectRename to the store

Goal

Approach

  • Covers R3. Committing a blank/whitespace name does not call update and restores the previous name.

Register rename + type-switch hotkeys
