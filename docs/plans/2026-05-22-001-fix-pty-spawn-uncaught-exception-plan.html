Fix: PTY spawn crashes main process with "Cannot create process, error code: 267"

Plan: docs/plans/2026-05-22-001-fix-pty-spawn-uncaught-exception-plan.html Branch: fix/pop-up-error Type: fix Depth: Standard crash bug

1. Problem Frame

Op Windows verschijnt onder bepaalde omstandigheden de Electron systeem-dialog:

A JavaScript error occurred in the main process
Uncaught Exception:
Error: Cannot create process, error code: 267
    at WindowsPtyAgent._completePtyConnection
    at EventEmitter2.fire
    at Worker.<anonymous>
    at MessagePort.<anonymous> (node:internal/worker:331:53)

Na OK klikken kan de app stuk zijn of in een halve staat doorlopen. Dit is een zichtbare, vertrouwen-erodende crash.

Scope

  • In scope: elimineer de uncaught exception bij pty.spawn, zowel root-cause (ongeldige cwd) als safety-net (global handler).
  • Out of scope: rewrite van session-restore flow, UI om "kapotte" projecten te herstellen, node-pty upgrade (los traject).

2. Root Cause Analysis

Windows error 267 = ERROR_DIRECTORY — "The directory name is invalid". Wordt teruggegeven door CreateProcessW wanneer lpCurrentDirectory niet bestaat, geen directory is, of niet bereikbaar is.

De keten naar de pop-up

#LocatieWat er gebeurt
1 electron/main/index.ts:457-505 (terminal:create IPC) Valideert ID-vormen, kiest cwd uit project/worktree. Geen disk-existence check in deze IPC-pad (de session-restore pad checkt wél, lijn 234-238).
2 electron/main/services/TerminalManager.ts:130-136 pty.spawn(shell, [], { cwd, env, ... }) — synchroon, geen try/catch, geen pre-validatie.
3 node-pty WindowsPtyAgent._completePtyConnection Roept CreateProcessW aan op een worker thread. Bij niet-bestaande cwd → error 267.
4 Worker.emit('error', ...) via MessagePort Error wordt async terug naar de main-thread EventEmitter gefired.
5 electron/main/index.ts Geen process.on('uncaughtException', …) handler. (grep bevestigt: 0 matches.) Electron toont default crash-dialog.

Waarom cwd kan verdwijnen tussen registratie en spawn

  • Worktree is buiten de app verwijderd (git worktree remove, rm -rf, IDE).
  • Project folder is hernoemd of verplaatst sinds toevoegen aan projects.json.
  • Externe / netwerk drive is ontkoppeld.
  • UNC path of pad met rare karakters dat CreateProcessW afwijst.
  • Race: worktree wordt verwijderd terwijl restore loopt (validatie passeert, spawn faalt milliseconden later).

Conclusie

Eén onveilige call (pty.spawn zonder pre-flight check), één ontbrekende vangnet (uncaughtException handler), en één inconsistente IPC-pad (alleen session-restore valideert cwd). Alle drie zijn klein en lokaal te fixen.

3. Solution Approach

Drie lagen van defense, van specifiek naar generiek:

Laag 1 — Voorkomen primair

cwd valideren vóór pty.spawn. Bij missende cwd: gooi een typed SpawnError die de IPC handler vangt en als gebruikersvriendelijke foutmelding terugstuurt — geen crash.

Laag 2 — Inkapselen secundair

Zelfs als pre-flight passeert (race, UNC, perms): wrap pty.spawn in try/catch én vang async pty error events op. Nooit een unhandled error vanuit de pty laag.

Laag 3 — Vangnet safety net

Globale process.on('uncaughtException') + 'unhandledRejection' handler in main/index.ts. Logt naar bestand, stuurt event naar renderer, voorkomt de system-dialog. Generic — vangt ook toekomstige onbekende failure-modes.

Laag 4 — UX optioneel

Renderer toont een toast/inline error in de terminal-tab: "Working directory bestaat niet meer — wil je het project verwijderen of het pad updaten?" — Buiten scope van deze fix, opgenomen onder Deferred.

4. Key Technical Decisions

BeslissingKeuzeWaarom
Waar valideren? In TerminalManager.createTerminal (single source of truth) plus early-return in terminal:create IPC voor betere UX. De manager wordt vanuit 2 plekken aangeroepen (IPC + session restore). Validatie in de manager garandeert dat elke call-site beschermd is. De IPC-laag voegt een sneller, vriendelijker error-pad toe.
Sync vs async check fs.statSync in manager (sync, snel, geen race-window tussen check en spawn). pty.spawn zelf is sync; een async pre-check zou een TOCTOU race introduceren met externe rmdir. De check moet zo dicht mogelijk vóór spawn staan.
Error type Nieuwe class SpawnError extends Error met code: 'CWD_MISSING' | 'CWD_NOT_DIR' | 'SPAWN_FAILED'. Typed errors zodat renderer per code een passende UX kan tonen. Sluit aan op bestaande convention (terminal:exit events).
Vangnet output Log naar userData/crash.log, stuur app:uncaught-error event naar renderer, géén app.exit(). Behoud van de andere terminals. Een uncaughtException van een pty worker is geen reden om alle Claude-sessies van de gebruiker te killen.
Bestaande session-restore check Laten staan, niet dupliceren. Geeft betere logging (console.log('worktree path X no longer exists')) vóór de IPC pas. Manager-validatie is de garantie; restore-validatie is de optimalisatie.

5. Implementation Units

U1. Pre-flight cwd validation in TerminalManager.createTerminal

Goal: Voorkom dat pty.spawn ooit met een niet-bestaande cwd wordt aangeroepen — root-cause fix.
Files:
  • electron/main/services/TerminalManager.ts (modify)
  • electron/main/services/errors.ts (new — kleine bestand voor SpawnError)
  • test/TerminalManager.spawn-validation.test.ts (new)
Approach:
  • Export class SpawnError extends Error met code en cwd velden uit errors.ts.
  • In createTerminal, vlak vóór de pty.spawn call (lijn ±130): statSync(cwd) in try/catch. Gooi SpawnError('CWD_MISSING', cwd) als ENOENT, 'CWD_NOT_DIR' als !stat.isDirectory().
  • Wrap de pty.spawn call zelf óók in try/catch → SpawnError('SPAWN_FAILED', cwd, { cause }). Vangt UNC/perm/race issues op.
  • Op de ptyProcess: registreer onError wanneer beschikbaar; anders verlaat je je op onExit + de spawn-tijd try/catch (node-pty exposes geen publiek onError — async errors uit de worker komen via process exit). Documenteer in comment.
Test scenarios:
  • Spawn met een bestaand pad → terminal-ID terug, geen error.
  • Spawn met niet-bestaande cwd → throws SpawnError met code === 'CWD_MISSING', geen pty process gecreëerd, geen registratie in terminals map.
  • Spawn met cwd die een file is (geen directory) → throws SpawnError met code === 'CWD_NOT_DIR'.
  • Spawn met geldige cwd maar shell-pad ongeldig (mock) → throws SpawnError met code === 'SPAWN_FAILED'.
  • Bij failure: hookWatcher.registerTerminal wordt niet aangeroepen, geen lekkende state in terminalInputBuffers / terminalTitled.
Verification: Vitest groen; handmatig: hernoem een project-folder, klik "new terminal" → inline error in plaats van crash-dialog.

U2. IPC handler vertaalt SpawnError naar renderer-event

Goal: Gebruiker ziet een nette inline melding, geen unhandled IPC rejection.
Dependencies: U1
Files:
  • electron/main/index.ts (modify terminal:create handler ±lijn 457-505 + session-restore call site ±lijn 264-274)
  • electron/preload/index.ts (modify — niets, IPC blijft hetzelfde) no change
  • src/types/index.ts (modify — voeg 'terminal:spawn-failed' event toe aan electronAPI type)
  • src/utils/terminalEvents.ts (modify — dispatch handler)
Approach:
  • Wrap de createTerminal call in terminal:create in try/catch. Bij SpawnError: stuur terminal:spawn-failed event met { projectId, worktreeId, code, cwd, message }; throw alleen voor IPC-validatie errors (zoals nu).
  • Session-restore loop: catch (error) bestaat al (lijn 293) — voeg discriminatie toe, log SpawnError verschillend (gebruiker krijgt geen modal bij restore, alleen log + skip).
  • Renderer luistert via terminalEvents.ts en toont een toast/banner.
Test scenarios:
  • IPC terminal:create met project waarvan pad is verwijderd → renderer ontvangt terminal:spawn-failed event, geen promise rejection, geen system-dialog.
  • Session-restore met cwd die net buiten de validatie window verwijderd is → loop slaat sessie over, console-log bevat SpawnError, andere sessies herstellen normaal.
Verification: E2E (Playwright): start app met fake project waarvan pad ontbreekt, klik new terminal, assert toast tekst.

U3. Global uncaughtException + unhandledRejection safety net

Goal: Nooit meer de Electron system-crash-dialog, zelfs niet bij onbekende failure-modes.
Dependencies: none (kan parallel met U1/U2)
Files:
  • electron/main/index.ts (modify — vroeg in app.whenReady, of nog beter top-level)
  • electron/main/services/CrashLogger.ts (new — klein, append-only)
Approach:
  • Registreer handlers vóór app.whenReady zodat ze ook vroege bootstrap errors vangen.
  • CrashLogger.log(error, { source: 'uncaughtException' | 'unhandledRejection' }) schrijft naar app.getPath('userData')/crash.log met timestamp + stack + Electron/Node versies.
  • Stuur app:uncaught-error event naar renderer met sanitized message (geen filesystem paths uit stacktrace). Renderer toont discrete toast: "Er is een interne fout opgetreden. Details in crash.log." met "Open log"-knop.
  • Geen app.exit(). De pty worker heeft één terminal kapot gemaakt; de andere blijven werken.
  • Rate-limit: max 5 log writes per minuut, anders log-spam bij een herhalende error.
Test scenarios:
  • Unit test: CrashLogger.log schrijft correct geformatteerde regels, rate-limit kicked in na 5 calls.
  • Manual: gooi setTimeout(() => { throw new Error('test') }, 100) in main → geen system-dialog, wel toast, wel crash.log regel.
  • Manual: reproduceer het originele scenario (delete project folder, klik new terminal vóór U1 patches) → safety net vangt het, app blijft draaien.
Verification: Manuele reproductie met origineel scenario; check crash.log; assert dialog blijft weg.

U4. Renderer toast/banner voor spawn-failures

Goal: Gebruiker begrijpt wat er mis is en weet wat te doen.
Dependencies: U2
Files:
  • src/components/notifications/SpawnErrorToast.tsx (new)
  • src/App.tsx (modify — listener registreren)
  • src/stores/projectStore.ts (modify — geen state; alleen kleine notification queue als die nog niet bestaat)
Approach:
  • Eén kleine component, gestyled met Tailwind, gebruikt bestaande theme CSS variabelen.
  • Mapping van SpawnError.code naar tekst:
    • CWD_MISSING → "De working directory bestaat niet meer: {cwd}. Verwijder het project of pas het pad aan."
    • CWD_NOT_DIR → "Het pad {cwd} is geen directory."
    • SPAWN_FAILED → "Kon geen shell starten in {cwd}. Zie crash.log."
  • Auto-dismiss na 8s, "Open log"-actie voor SPAWN_FAILED.
Test scenarios:
  • Component renders met elke code variant.
  • E2E: trigger spawn-failure → toast verschijnt → 8s later weg.
Verification: Manueel in dev-server; visuele check.

6. System-Wide Impact

  • Performance: één extra statSync per terminal-create — verwaarloosbaar (microseconden).
  • Bestaande sessions-restore flow: validatie blijft, wordt nu een snelle pad; SpawnError in restore wordt zachter behandeld dan in user-initiated create.
  • Logging: nieuwe crash.log in userData/. Documenteer in support-paragraaf van README.
  • Bestaande tests: TerminalPool + projectStore unit tests onaangetast. Nieuwe tests breiden coverage uit.

7. Risks & Mitigations

RiskLikelihoodMitigation
node-pty gooit ook andere worker-errors die we per ongeluk swallowen. Laag Safety net logt altijd volledig stacktrace; rate-limit is hoog genoeg om iteratieve debug toe te staan.

Toast verschijnt in een achtergrond-window dat gebruiker niet ziet. Laag Notification queue bewaart laatste 3 errors; ook zichtbaar via een history-icoon (Deferred — niet in deze PR).
uncaughtException handler te vroeg geregistreerd, voordat logger pad weet. Laag CrashLogger gebruikt app.getPath('userData') lazy bij eerste log(); valt terug op os.tmpdir() als app niet ready is.

8. Scope Boundaries

Deferred to follow-up

  • "Repair project" UX: knop op het project in de sidebar om pad opnieuw te kiezen wanneer cwd ontbreekt.
  • Notification history-panel.
  • Automatische detectie van verdwenen worktrees bij app-start (proactieve cleanup).

Niet onderdeel van dit fix

  • node-pty versie-upgrade.
  • Vervangen van de Electron crash-dialog door een custom in-app dialog voor andere uncaught errors (zou kunnen, niet nodig voor deze fix).
