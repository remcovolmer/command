Goal Capsule

Objective
Relocate automations to a global left-sidebar entry with a full-area management overview (decoupled from the active project), and add a focus-neutral foreground launch that spawns a live chat or worktree in the automation's project. Execution stays coupled to exactly one project per automation.
Product authority
Remco (owner)
Open blockers
None. Two runtime edges are carried as deferred implementation detail (very-long-prompt command-line limit; prompt shell-quoting) — see Risks.

Product Contract

Summary

Promote automations to a global, top-level surface: a sidebar entry above the projects list that opens a flat-table overview of every automation, independent of the active project. Each automation belongs to exactly one project and can be launched on demand into a live, permission-gated chat or worktree (foreground — the primary use), or run headless on a trigger (secondary). Launches never steal focus; a spawn animation signals the new session. One run-history timeline covers both kinds of run.

Problem Frame

The automation data model is already global — an automation stores its own project associations and the panel already lists every automation. But the only UI lives inside the right-side file-explorer flyout, behind an activity-rail icon. That surface reads as belonging to the active project, so managing automations across projects means hunting through a per-project flyout, and there is no single place to see everything.

Runs are headless-only: a claude -p process in a throwaway worktree whose output you read after the fact. You cannot launch a saved automation as a live, interactive session and let it run while you keep working. The net effect is that automations feel like a hidden, fire-and-forget corner rather than a first-class way to launch work.

Key Decisions

Launch-template model, not a cron manager

On-demand foreground firing is the primary interaction; scheduled/event triggers are secondary. The overview is a launcher-with-management, not a schedule dashboard.

Single project per automation

Each automation belongs to exactly one project, replacing today's many-to-many projectIds[]. Simplest data model and UI; reuse means duplicating a template.

Foreground is an interactive, permission-gated chat

A foreground launch behaves like a normal chat: it runs under the project's Claude mode and prompts for permissions, unlike headless runs which skip permissions.

Focus-neutral launch

No automation-initiated run — manual or triggered — changes the active project or terminal. The spawned session appears in the sidebar, selectable, but never interrupts what you're doing. A spawn animation plus an origin marker make it noticeable.

Interface decoupled, execution coupled

The overview is global and does not depend on the active project. Every run still executes against exactly one project.

Move, not duplicate

The Automations tab is removed from the right-rail flyout and the activity rail entirely; the overview is the single home for automations.

Key Flows

F1. Foreground launch. Overview → row Launch → optionally override target for this run → a chat spawns (project working dir or a new worktree) started via claude "<prompt>", running under the project's Claude mode → it appears in the sidebar with a spawn animation and origin marker, without switching focus → run history logs an entry linked to that session.

F2. Headless triggered run. A schedule or event trigger fires → the runner creates a throwaway worktree → claude -p runs headless with captured output → run history logs an entry with output. No visible chat, no focus change. (Unchanged from today.)

F3. Manage. Overview → create or edit an automation → set its single project, trigger, default target, and prompt → enable/disable via the row toggle. All from the global overview, without touching the active project.

Acceptance Examples

AE1. Foreground launch, worktree default, no override. Covers R12, R13, R16, R21, R22, R20.

An automation whose default target is a worktree is launched from its row. A new persistent worktree is created and a chat spawns inside it via claude "<prompt>" under the project's Claude mode; permission prompts apply when the session hits a gated tool. The active project/terminal do not change; the new worktree+chat appear in the sidebar with a spawn animation and origin marker, and a run-history entry links to that session.

AE2. Per-run target override. Covers R14, R15.

At launch the user overrides a chat-default automation to run in a worktree for this one launch. The run uses the worktree target; the automation's stored default remains "chat" for the next launch.

AE3. Scheduled trigger fires while the user works. Covers R17, R18, R19, R20, R21.

A cron trigger fires mid-task. The run executes headless in a throwaway worktree, output is captured, and a run-history entry shows it. No visible chat opens, the active project/terminal are untouched, and the throwaway worktree is auto-GC'd per existing rules.

AE4. Migrating a multi-project automation. Covers R10, R11.

An existing automation associated with two projects is loaded after the change. It is bound to the first associated project; the second association is dropped, and the overview shows it under that one project.

AE5. Open the overview while another project is active. Covers R3, R5.

Project B is active with open chats. Opening Automations shows every automation across all projects; project B stays active and its chats/worktrees are untouched. Leaving the overview returns to project B unchanged.

Scope Boundaries

Outside this change
  • Trigger types stay as they are — no new trigger kinds (cron / claude-done / git-event / file-change unchanged).
  • The headless execution mechanism, concurrency cap, and worktree GC lifecycle are unchanged.
  • No cross-project reuse of a single automation, no chaining, no import / export / sharing.
Deferred to follow-up work
  • Temp-file / stdin fallback for prompts that exceed the OS command-line length limit (basic positional-arg path ships in U3; the fallback is follow-up unless the simple path proves insufficient).
  • Cron/event triggers that spawn a foreground chat instead of headless — triggers stay headless for now.
  • Configurable spawn animation and richer run-history filtering.

Dependencies & Assumptions

Assumptions
  • claude "<prompt>" starts an interactive session with the prompt auto-submitted (verified via claude --help: "starts an interactive session by default", positional [prompt]).
  • The project's Claude mode is already resolved into the standard chat-create path (handleTerminalCreate → TerminalManager.createTerminal's claudeMode option), so a foreground launch inherits permission-gating without new wiring.
  • Multi-project automations are rare in practice, so migrating to the first associated project is acceptable without a per-automation prompt.

Planning Contract

Key Technical Decisions

KTD1.Launch via positional claude "<prompt>", not timed PTY injection

The existing PTY writes the claude command after a shell-ready delay; a foreground launch appends the automation prompt as the positional argument so the interactive session starts with the prompt already submitted. This removes any need to detect REPL readiness before injecting keystrokes. The prompt must be shell-quoted safely; very long prompts risk exceeding OS argument limits (see Risks, temp-file fallback deferred).

KTD2.Single-project data model with in-place migration

Automation.projectIds: string[] becomes projectId: string plus a new defaultTarget: 'chat' | 'worktree'; AutomationRun gains mode: 'foreground' | 'headless' and terminalId?: string. AutomationPersistence bumps its state version and migrates on load: first projectIds entry → projectId, missing defaultTarget → worktree, existing runs → headless.

KTD3.Focus-neutral launch, compensated by a spawn cue

The launch path never calls setActiveProject/setActiveTerminal. Because focus stays put, the spawned sidebar item plays an appearance animation and carries an origin marker (a new origin: 'automation' field on TerminalSession) so the user notices. This rule is universal — manual launches and any triggered run.

KTD4.Overview as a project-independent center overlay

A new automationsOverviewVisible store flag mirrors projectOverviewVisible. TerminalArea renders <AutomationsOverview /> at the top of its render — before the "no active project" welcome branch — so the overview shows regardless of active project (R3, R5, AE5). The flag is not cleared by a launch (launch is focus-neutral).

KTD5.Foreground orchestration is renderer-side; main only records the run

A useLaunchAutomation hook resolves project + target (+ override), creates the worktree when needed (reusing the existing worktree-create path), creates the chat via useCreateTerminal with an initialPrompt and without activating it, then calls a new automation:record-launch IPC so the main process logs a foreground run linked to the terminal. The headless AutomationRunner is untouched.

KTD6.Move, not copy — delete the panel

AutomationsPanel.tsx is deleted; its markdown-render/status-icon/formatter helpers move into the new overview. The activity-rail entry, the ExplorerTab 'automations' member, and the file-explorer tab case are removed.

High-Level Technical Design

System-Wide Impact

  • Persisted data migration. automations.json and automation-runs.json migrate on first load (new state version). One-way; safe defaults.
  • IPC contract. terminal:create gains an initialPrompt parameter; a new automation:record-launch channel is added. Preload + renderer types update in lockstep.
  • Layout. A new sidebar entry and a new center overlay; the right-rail activity icon and file-explorer tab are removed.
  • Hotkey + docs. Ctrl+Shift+A rebinds from "automations tab" to "show automations overview"; the CLAUDE.md shortcuts table updates.

Risks & Mitigations

Risk · prompt quoting
The automation prompt is interpolated into a shell command line written to the PTY. Unsafe quoting could break the command or allow injection. Mitigation: quote the prompt as a single argument with proper escaping; unit-test the command builder with quotes, newlines, backticks, and $.
Risk · very long prompts
The dialog allows prompts up to 50,000 chars; a positional argument that long can exceed OS command-line limits. Mitigation: ship the positional-arg path for normal sizes; carry a temp-file/stdin fallback as deferred follow-up and cap or warn if the limit is approached.
Risk · migration data loss
Collapsing projectIds[] to a single projectId drops extra associations. Mitigation: accepted per R11; migration keeps the first project and is covered by a migration test against a v1 fixture.
