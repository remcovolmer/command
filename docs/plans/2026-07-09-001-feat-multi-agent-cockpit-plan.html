Goal Capsule

Objective. Codex en pi.dev toevoegen als eerste-klas agents in Command, met best-effort attention-state per agent, zodat één cockpit meerdere coding-agents naast elkaar draait — Claude niet langer als enige.

Product authority. Remco (owner Command). Product Contract komt uit de brainstorm-fase en blijft leidend voor het wat.

Stop conditions. Stop en overleg bij: een wijziging die Claude-chats zou laten regresseren (state, resume, auto-naming); een codex/pi CLI-contract dat materieel afwijkt van wat dit plan aanneemt (andere hook-payload, ander resume-mechanisme); of scope-uitbreiding voorbij codex+pi.

Product Contract preservation. Ongewijzigd — dit plan verrijkt hetzelfde artifact met de hoe (Planning Contract, Implementation Units, Verification, Definition of Done); R-IDs en Product-Contract-tekst blijven intact.

Volgorde. Codex eerst end-to-end (Phase 1 = fundament + codex), pi als dunne toevoeging (Phase 2). Pi's spawn komt gratis uit de descriptor; alleen de busy/done-heuristiek is pi-specifiek.

Product Contract

Summary

Command wordt agent-agnostisch. TerminalType krijgt naast claude en normal ook codex en pi, elk gespawnd als PTY-sessie via het eigen CLI-commando. De attention-indicator werkt best-effort per agent: heeft de agent een hook-mechanisme, dan volle staten (busy/permission/question/done); zo niet, dan busy/done. Codex en pi zijn in v1 hardcoded — geen generieke registry — maar per-agent gedrag staat op één plek (een AGENTS-descriptor), zodat die registry later een refactor is, geen herbouw.

Key Decisions

KD1Command wordt agent-agnostisch, niet Claude-met-plug-ins

Codex en pi zijn volwaardige alternatieven, geen add-ons op Claude. Herpositionering van "Claude-Code-manager" naar cockpit voor coding-agents. Driver: agent-diversiteit als principe.

KD2Best-effort state per agent verslaat geforceerde pariteit

Hook aanwezig → volle staten; anders busy/done. Elke agent doet mee zonder dat de zwakste (pi) de lat voor de sterkste (codex) verlaagt.

KD3Hardcoded codex + pi, maar per-agent gedrag gecentraliseerd

Geen generieke config-registry vooraf. Wél: spawn/resume/state/badge/mode per agent in één AGENTS-descriptor. Snelste weg naar waarde zonder het "agnostisch"-fundament te verraden.

KD4Wrap de CLI, niet de SDK

Codex en pi worden PTY-sessies zoals Claude — consistent met Command's terminal-first-architectuur. Geen embedded agent-runtime.

High-Level Concept — gedeelde state-laag

De kern-refactor: alle agents voeden dezelfde attention-indicator via één gedeelde state-file. De watcher hoeft niet te weten wélke agent een entry produceerde — hij mapt op session_id. Codex sluit hier naadloos op aan (identiek hook-contract als Claude); pi voedt busy/done via heuristiek.

Key Flows

  1. F1  Nieuwe agent-chat aanmaken (sidebar-first)

    • Trigger: gebruiker klikt de per-project + in de sidebar (of maakt een worktree aan).
    • Command pakt de project-default-agent (R3); een picker op de + laat eenmalig overriden (R4). Worktree-chats erven de project-default.
    • PTY spawnt het CLI-commando van de gekozen agent (R2); de chat-tab toont de agent-badge (R5); state start op busy.

  2. F2  Codex state-lifecycle (volle staten)

    • Trigger: codex vuurt een hook-event (SessionStart, PreToolUse, PermissionRequest, Stop, …).
    • Het codex-hook-script schrijft {session_id, cwd, state, event, ts} naar de gedeelde state-file (R7, R8).
    • De watcher pollt, mapt session→terminal, emit terminal:state → indicator (busy/permission/done).

  3. F3  Pi state-lifecycle (best-effort)

    • Trigger: pi-chat produceert output of valt stil.
    • Geen config-hook → Command leidt busy/done af uit PTY-output-activiteit (R6, R9).
    • Indicator toont busy/done; permission/question niet apart voor pi in v1.

Agent-keuze & badges (wireframe, sidebar-first)

Acceptance Examples

AE1  Covers R6, R8. Een codex-chat roept een tool aan → indicator wordt busy. Codex vraagt approval → permission. De turn stopt → done.

AE2  Covers R6, R9. Een pi-chat produceert output → busy. Pi is ≥ de stil-drempel inactief → done. Permission/question niet apart voor pi.

AE3  Covers R3, R4. Project-default = codex, sidebar-+ zonder override → codex-chat. Met override = pi → pi-chat. Beide tonen de juiste badge.

AE4  Covers R10, R11. Command herstart met een openstaande codex-chat → hervat via codex' resume-mechanisme (niet vers), en de codex-sessie-ID passeert validatie.

Success Criteria

  • Je start vanuit de sidebar een codex- én pi-chat naast Claude, in dezelfde cockpit.
  • Codex-sessies tonen de volledige attention-staten; pi-sessies minimaal busy/done.
  • Per chat zie je direct welke agent draait (badge op sidebar-rij én tab).
  • Geen regressie op Claude-chats: state, resume en auto-naming blijven werken.

Scope Boundaries

Deferred for later — niet v1
  • Generieke, config-toevoegbare agent-registry (elke CLI zonder code).
  • Volle permission/question-pariteit voor pi (extensie-API-spike).
  • 1:1 mapping van Command-modes (chat/auto/full-auto) naar codex/pi permission-modes.
  • Auto-naming voor codex/pi — nice-to-have (zie U2-notitie).
Outside this product's identity
  • SDK/API-integratie van agents — Command blijft terminal-first: wrap de CLI, geen embedded runtime.
  • Een eigen agent/model bouwen — Command orkestreert bestaande CLI-agents.

Planning Contract

Key Technical Decisions

KTD1Verbreed TerminalType i.p.v. een apart agent-veld

type loopt al end-to-end door IPC (terminal:create arg 3), preload, store en UI. Verbreden naar 'claude'|'codex'|'pi'|'normal' is de laagste frictie: de !== 'normal'-filters blijven kloppen; alleen de === 'claude'-filters verbreden. Terminals worden niet gepersisteerd (partialize in src/stores/projectStore.ts laat terminals weg) → geen store-migratie voor de sessie-shape.

KTD2Eén gedeelde state-file; watcher blijft agent-agnostisch

Alle agent-hooks schrijven naar ~/.claude/command-center-state.json, keyed op session_id. De watcher leest die al zo — het mechanisme is al agnostisch; alleen de registratie (nu claude-only in TerminalManager) verbreedt naar hook-capabele agents.

KTD3Codex hergebruikt het Claude-hookcontract ~1:1

Kloon electron/main/hooks/claude-state-hook.cjs → codex-state-hook.cjs. Codex-events (SessionStart/PreToolUse/PermissionRequest/Stop) mappen op dezelfde staten; payload heeft session_id+cwd. Installeer in ~/.codex/hooks.json via een gegeneraliseerde HookInstaller.

KTD4Centrale AGENTS-descriptor als enige per-agent-naad

Eén map met per agent: binary, resume-flag-builder, mode-flags, hasHook/hook-config, en (renderer-zijde) badge/label. Spawn, resume, install en UI lezen hieruit. De latere generieke registry = deze map vervangen door config-geladen entries (KD3).

KTD5Pi-state = output-heuristiek; extensie-spike deferred

Voor pi (geen config-hook): busy op PTY-output, done na een stil-timeout. Permission/question niet gesurfaced in v1. Als de pi-extensie-API lifecycle-events blijkt bloot te leggen, kan een pi-extensie later naar dezelfde state-file schrijven voor vollere staten — dat is een aparte spike, geen v1-blocker.

KTD6Sessie-persistence generaliseert weg van Claude-hardcoding

PersistedSession krijgt een agentType; verifyClaudeSessionAsync wordt per-agent transcript-pad; restoreSessions gebruikt de opgeslagen agent i.p.v. hardcoded type: 'claude'; de before-quit-save en getClaudeTerminalIds dekken alle agent-chats. Resume-commando komt uit de descriptor.

Unit-afhankelijkheden & volgorde

Assumptions

  • Codex-hooks (geverifieerd): stdin-JSON met session_id+cwd+hook_event_name; install-pad ~/.codex/hooks.json idempotent te mergen náást bestaande gebruikersconfig, net als de Claude-installer.
  • Pi-events (onbevestigd): geen gedocumenteerd config-hook-mechanisme; v1 = output-heuristiek. De extensie-API-spike is deferred.
  • Agents geïnstalleerd: codex en pi op PATH; Command spawnt de CLI maar installeert die niet.
  • Terminals niet gepersisteerd: alleen ProjectSettings (incl. nieuwe defaultAgent) migreert — geen sessie-shape-migratie nodig.

Open Questions

Deferred to implementation
  • Exact codex session_id-formaat en hoe de UUID-validatie in de hook/persistence per agent generaliseert (regex verruimen vs per-agent validator).
  • Idempotente install én de-install van de codex-hook náást bestaande codex-config; en of Command de codex-hook bij afsluiten opruimt.
  • Pi stil-timeout-drempel voor de busy/done-heuristiek (start ~1,5–2s; kalibreren tijdens smoke).
  • Codex resume-invocatie: exacte flag/subcommando (codex resume vs --resume) verifiëren tegen de geïnstalleerde CLI-versie in U6.
