Render the bottom shell drawer for type: 'project' ("Files + Claude") folders, and hide the now-phantom git icon from the activity rail for them — without turning on git or any other Code-only tooling.

Goal Capsule

Objective. Let folders added as the lightweight 'project' type use the bottom shell drawer exactly as 'code' projects do, and remove the leftover git affordance (the rail icon) that contradicts those folders' "no git" identity.

Authority hierarchy. Confirmed scope (shell only; git stays off) > this plan > repo conventions. Do not collapse the two project types or enable other Code-only tooling.

Stop conditions. Stop and surface if enabling the shell drawer would require touching the git tab's content gating, the PTY spawn path, or session-restore logic — none of those should be in this change. Stop if hiding the git rail icon breaks the rail for 'code' projects.

Execution profile. Small, low-risk renderer-only change across three files plus two component tests. No main-process, IPC, or persistence changes.

Product Contract

Summary

FleetView projects come in two types: 'project' (labelled "Files + Claude", the lightweight "limited" type) and 'code' (full dev tools). The bottom shell drawer — the per-context plain-shell panel toggled from the activity-rail foot and Ctrl + \ — is suppressed for 'project' folders by a single render guard. This plan removes that guard so the shell drawer works for those folders, and hides the git icon from the activity rail for them so the surface matches the "no git" decision. Git tab content and all other Code-only tooling stay off.

Problem Frame

Three things are wrong for a 'project'-type folder today, all stemming from the shell being half-wired:

  • The shell drawer never renders — ShellDrawer returns null when the active project's type === 'project'.
  • The rail shell toggle and Ctrl + \ are not gated, so triggering them calls createSidecarTerminal and spawns a real PTY that the drawer then refuses to show — an invisible, unclosable shell.
  • The git icon still appears in the activity rail, but clicking it just opens the file-explorer flyout on the Files tab (the explorer forces git → files for limited projects). A control that resolves to something other than its label.

Planning Contract

Key Technical Decisions

KTD1  Remove the gate, don't invert it. The store action (createSidecarTerminal), the shell.toggle hotkey, the rail toggle, and the main-process PTY spawn are all already type-agnostic. The render guard in ShellDrawer is the single source of the restriction, so deleting it (and its now-unused projects / activeProject / isLimitedProject lookups) is the whole behavior change for the shell. This also incidentally fixes the invisible-shell dead state (R2), because the already-spawned PTY now renders.

KTD2  Hide git from the rail by filtering its item list; keep the explorer fallback. Filter out the git entry in ActivityRail when the active project's type === 'project'. Leave FileExplorer's existing activeTab === 'git' ? 'files' coercion in place as a defensive net — it still guards persisted/stale state (e.g. a folder whose last tab was git before this change, or a code-then-project edge). Removing both at once would be the only way a limited project could show git content, so the fallback stays.

KTD3  Hold the limited-type identity. Only the shell is enabled and only the git icon is removed. Tasks, Automations, and Files rail entries are untouched. Full parity with the 'code' type is explicitly not pursued, so the two types remain distinct.

High-Level Technical Design

The two coupled changes both make the 'project'-type surface consistent: turn the shell on, take the phantom git control off. Before/after for a 'project' folder:

Before

Files Git Tasks Automations Shell ⌗
Git icon present, but click → flyout opens on Files (label lies) Shell toggle / Ctrl+\ → createSidecarTerminal → PTY spawns → ShellDrawer returns null → invisible, unclosable shell

After

Files Git (removed) Tasks Automations Shell ⌗
No git icon — rail matches "no git" identity Shell toggle / Ctrl+\ → createSidecarTerminal → PTY spawns → ShellDrawer renders → visible shell with tabs

Git tab content stays disabled in both states; only the rail entry point is removed. The explorer's git → files coercion (KTD2) keeps stale state safe even with the icon gone.

Approach
  • Change the early return from if (!contextKey || !activeProjectId || isLimitedProject) return null to if (!contextKey || !activeProjectId) return null.
