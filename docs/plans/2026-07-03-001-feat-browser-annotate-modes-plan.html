Goal Capsule

Objectief: voeg drie annotatie-modi toe aan de ingebouwde browser-webview zodat de gebruiker vanuit een gerenderde pagina (dev-server of lokaal bestand) context naar de actieve Claude-chat stuurt — door te selecteren + becommentariëren, inline te editen, of te tekenen + een screenshot te sturen. De agent past de bron aan; de gebruiker ververst.

Product-autoriteit: Remco. Origin: de requirements-only brainstorm docs/brainstorm/2026-07-02-browser-annotatie-command.html (Product Contract, R1–R13, KD-A t/m E).

Product Contract preservation

Product Contract ongewijzigd. Eén afwijking t.o.v. de brainstorm: de hotkey-eis is uit scope gehaald (besluit gebruiker 2026-07-03) — hotkeys waren een repo-conventie, geen Product Contract-item. De CLAUDE.md-regel is bijgewerkt naar "kernfunctionaliteit/primaire workflows".

Verankerd (KD1, 2026-07-02)

Terminal-first. De webview is een content-surface, geen chat-shell; annotatie is een input-methode die via terminal.write in de bestaande Claude-chat landt — nooit via SDK/API.

Planning Contract — Key Technical Decisions

KTD1 Host-pull architectuur; primaire UI in de React-toolbar. Command's renderer stuurt de guest via executeJavaScript/capturePage op de webview-ref. Mode-switcher, comment-invoer en actieknoppen leven in de bestaande React browser-toolbar (buiten de webview → geen z-index/paint-conflict). Alleen de highlight en het teken-canvas worden ín de guest geïnjecteerd (moeten wel: over de pagina liggen + in capturePage zitten). Lost KD-A én de selectie-UX uit de brainstorm op.

KTD2 Breid CommandWebviewElement uit met executeJavaScript en capturePage. Het is een echte Electron WebviewTag (webviewTag: true in electron/main/index.ts); beide methodes werken native en ontbreken alleen in src/types/webview.d.ts ("we declare only the surface BrowserTab drives"). Geen Electron-side wijziging nodig.

KTD3 Guest→host is host-pull-only in de MVP. Triggers zijn toolbar-knoppen; op klik leest Command de guest-state via executeJavaScript (dat een Promise met de returnwaarde teruggeeft). Geen guest→host push-kanaal. Het console-message-kanaal (voor auto-popover-op-selectie, zonder de hardening te verzwakken) is een expliciete latere upgrade. Kosten nu: één extra klik (selecteer → knop).

KTD4 Modus 3-afbeelding via clipboard.writeImage + Alt+V. Nieuw IPC-kanaal dat clipboard:writeText spiegelt (een ipcMain.on in electron/main/index.ts, met een sanitizeClipboardImage-helper in electron/main/utils/clipboardLimits.ts en nativeImage). Claude Code plakt de klembord-afbeelding met Alt+V (Claude Code-capability, geen Command-code). Geen binaire file-write of temp-bestand.

KTD5 Grote payloads chunken — terminal.write kapt af rond ~1 KB. Bekend probleem (node-pty 1.1.x EAGAIN + ConPTY backpressure, zie docs/solutions/integration-issues/node-pty-paste-truncation.md). Modus 1/2 sturen outerHTML-snippets die dit overschrijden; het bericht wordt in stukken geschreven met de gedocumenteerde veilige write-cadans, niet in één keer.

KTD6 Source-mapping (uit KD-C). Lokaal bestand mapt direct naar zichzelf; voor een dev-server stuurt Command outerHTML + selector + DOM-pad + route mee en laat Claude de juiste component vinden. Deterministisch vs. agent-resolved.

Implementation Units

U1. Webview-control uitbreiden (executeJavaScript / capturePage)

Goal
De host in staat stellen JS in de guest uit te voeren en de pagina te capturen — het fundament voor alle modi.
Requirements
R13
Dependencies
—
Files
src/types/webview.d.ts, src/components/Browser/BrowserTab.tsx, test/webviewControl.test.ts
Approach · Patterns · Test scenarios · Verification

Approach: voeg executeJavaScript(code: string): Promise<unknown> en capturePage(): Promise<Electron.NativeImage> toe aan het CommandWebviewElement-interface (native op de WebviewTag, alleen niet gedeclareerd). Introduceer een dunne host-side helper (in BrowserTab of een klein hook) die deze aanroepen achter de bestaande readyRef-guard zet, zodat calls vóór dom-ready niet gooien.

Patterns to follow: ref-gebruik en readyRef-guard in BrowserTab.tsx (de bestaande goBack()/loadURL()-calls); de .d.ts-conventie "declare only what we drive".

Test scenarios:

  • Helper roept niet aan vóór dom-ready (readyRef false → no-op, geen throw).
  • executeJavaScript-helper geeft de returnwaarde van de guest-expressie door (gemockte webview-ref).
  • capturePage-helper resolve't naar een image-object (gemockt).

Verification: vanuit devtools/console een triviale executeJavaScript('document.title') op een geladen pagina geeft de titel terug.

U7. IPC clipboard:writeImage

Goal
Een afbeelding op het OS-klembord kunnen zetten, zodat modus 3 via Alt+V in de chat landt.
Requirements
R11
Dependencies
—
Files
electron/main/index.ts, electron/main/utils/clipboardLimits.ts, electron/preload/index.ts, src/types/index.ts, test/clipboardLimits.test.ts
Approach · Patterns · Test scenarios · Verification

Approach: nieuwe ipcMain.on('clipboard:writeImage', ...) die de payload (data-URL of buffer) door een sanitizeClipboardImage-helper haalt (size-cap + type-check, mirror van sanitizeClipboardText), dan clipboard.writeImage(nativeImage.createFromDataURL(...)). Voeg de nativeImage-import toe. Expose clipboard.writeImage(dataUrl) in preload (send) en in het ElectronAPI-type.

Patterns to follow: clipboard:writeText + sanitizeClipboardText/MAX_CLIPBOARD_TEXT_BYTES in clipboardLimits.ts; de { ok } | { reason }- of value | null-validatievorm; preload-namespace clipboard.

Test scenarios: Covers R11.

  • Geldige PNG-dataURL onder de cap → sanitizer geeft image terug.
  • Payload boven de size-cap → null (no-op), geen throw.
  • Niet-image / kapotte dataURL → null.
  • Niet-string / niet-verwacht type → null.

Verification: handler aanroepen met een kleine test-PNG → het OS-klembord bevat de afbeelding (of Alt+V in een chat plakt 'm).

U3. Bericht samenstellen + naar actieve Claude-chat schrijven

Goal
Eén plek die de context-rijke berichten bouwt en veilig (chunked) naar de actieve claude-terminal schrijft, met de "geen chat"-guard.
Requirements
R2, R3, R5, R6, R8, R9, R12
Dependencies
—
Files
src/utils/annotationMessage.ts, test/annotationMessage.test.ts, src/stores/projectStore.ts (lookup hergebruiken)
Approach · Patterns · Test scenarios · Verification

Approach: pure builders per modus (comment/edit → tekst met snippet/voor→na + locator + route). Een send-functie die de actieve claude-terminal opzoekt (activeTerminalId → terminals[id], type === 'claude'), en bij afwezigheid een guard-resultaat teruggeeft (R3). Grote payloads worden in stukken via terminal.write geschreven met de veilige cadans (KTD5).

Execution note: begin met een falende test voor de chunk-grens (payload > ~1 KB) — dat is de kern van KTD5.

Patterns to follow: store-lookup state.terminals[activeTerminalId] + type-check; getElectronAPI().terminal.write(id, data); de node-pty paste-truncation-oplossing in docs/solutions/.

Test scenarios: Covers R3.

  • Comment-builder produceert bericht met snippet + comment + route.
  • Edit-builder produceert "verander X → Y op <locator>".
  • Payload > ~1 KB → meerdere terminal.write-calls (chunked), niet één.
  • Geen actieve claude-terminal → guard-resultaat, geen write.
  • Actieve terminal is normal (geen claude) → guard-resultaat.

Verification: met een actieve chat verschijnt het samengestelde bericht in de terminal; zonder chat gebeurt er niets en meldt de UI dat.

U4. Modus 1 — Aanwijzen & commentaar

Goal
Selectie op de pagina + opmerking → snippet + comment naar de chat.
Requirements
R4, R5, R6
Dependencies
U1, U2, U3
Files
src/components/Browser/BrowserTab.tsx, src/utils/annotationGuestScript.ts, test/annotationGuestScript.test.ts
Approach · Patterns · Test scenarios · Verification

Approach: in comment-modus klikt de gebruiker de toolbar-knop "commentaar"; Command roept executeJavaScript aan met een expressie (uit annotationGuestScript) die de huidige window.getSelection() + omliggende outerHTML + een CSS-selector/DOM-pad + location.href als JSON teruggeeft, en een tijdelijke highlight om de selectie tekent. Een React comment-invoer (toolbar) neemt de opmerking; op verzenden bouwt en stuurt U3 het bericht.

Patterns to follow: U1-helper; U3-builders; toolbar-UI uit U2.

Test scenarios: Covers R4, R5, R6 / AE1.

  • Guest-script-builder produceert geldige JS die selectie + outerHTML + selector + href retourneert (unit-test op de gegenereerde string / tegen een jsdom-fragment).
  • Lege selectie → knop geeft nette "niets geselecteerd"-feedback, geen bericht.
  • Geldige selectie + comment → U3 krijgt snippet + comment + route (gemockt).
  • Highlight wordt gewist na verzenden of bij mode-wissel.

Verification: op een lokaal .html én op een localhost dev-app: selecteer tekst, typ een opmerking, verstuur → het bericht (met snippet + route) staat in de actieve chat.

U5. Modus 2 — Inline editen

Goal
Gehighlighte tekst ter plekke bewerken → voor→na als edit-instructie naar de chat.
Requirements
R7, R8, R9
Dependencies
U1, U2, U3
Files
src/components/Browser/BrowserTab.tsx, src/utils/annotationGuestScript.ts, test/annotationGuestScript.test.ts
Approach · Patterns · Test scenarios · Verification

Approach: in edit-modus zet Command via executeJavaScript het geselecteerde element (of een wrapper om de selectie) op contentEditable en legt de voor-tekst + locator vast. De gebruiker bewerkt in de pagina. Op "verstuur edit" leest Command de na-tekst terug en laat U3 een edit-instructie sturen. De pagina-edit is tijdelijk (niet gepersisteerd); contentEditable wordt daarna weer uitgezet.

Execution note: let op selecties die meerdere elementen overspannen — beperk de MVP tot een enkel tekst-dragend element en geef nette feedback bij een complexe selectie.

Patterns to follow: U1-helper; U3-edit-builder.

Test scenarios: Covers R7, R8, R9 / AE2.

  • Guest-script legt voor-tekst + locator vast bij start van de edit.
  • Na bewerken levert het na-lezen de gewijzigde tekst; ongewijzigd → geen bericht.
  • Edit-instructie bevat voor→na + locator (gemockte U3).
  • Complexe/multi-element selectie → nette feedback, geen kapot bericht.
  • contentEditable staat uit na verzenden/annuleren.

Verification: highlight een woord, corrigeer het inline, verstuur → de chat krijgt "verander X → Y op <locator>"; de pagina zelf blijft onveranderd tot de agent de bron aanpast.

U6. Modus 3 — Tekenen & snapshot

Goal
Vrij tekenen op de pagina + "voeg toe aan chat" → screenshot van pagina+tekening via het klembord naar de chat.
Requirements
R10, R11, R12
Dependencies
U1, U2, U3, U7
Files
src/components/Browser/BrowserTab.tsx, src/utils/annotationGuestScript.ts
Approach · Patterns · Test scenarios · Verification

Approach: in teken-modus injecteert Command via executeJavaScript een full-page canvas-overlay met pointer-handlers ín de guest (zodat de tekening in capturePage zit). Op de toolbar-knop "voeg toe aan chat" roept Command capturePage() aan, zet de afbeelding via clipboard.writeImage (U7) op het klembord, en stuurt via U3 een begeleidend tekstbericht met de route. De afbeelding landt via Alt+V (Claude Code-capability). De canvas wordt daarna gewist.

Execution note: Alt+V programmatisch triggeren (byte-sequence naar de PTY) is best-effort; fallback = de gebruiker drukt zelf Alt+V. Verifieer eerst dat capturePage de geïnjecteerde canvas meepakt.

Patterns to follow: U1 capturePage-helper; U7 clipboard-IPC; U3 voor het tekstdeel.

Test scenarios: Covers R10, R11, R12 / AE3.

  • Guest-script-builder produceert canvas-injectie + wis-routine (unit-test op de string / jsdom).
  • "Voeg toe aan chat" zonder tekening → nette no-op of feedback.
  • Met tekening → capturePage + clipboard.writeImage aangeroepen (gemockt) + tekstbericht met route via U3.
  • Canvas gewist na verzenden / mode-wissel.

Verification: op een dev-app: teken een cirkel om een element, klik "voeg toe aan chat", plak met Alt+V → de screenshot mét tekening verschijnt in de chat, met de route als tekst erbij.

Verification Contract

  • Unit tests (vitest, test/): clipboardLimits image-sanitize (U7), annotationMessage builders + chunk-grens + chat-guard (U3), annotationGuestScript builders (U4–U6), webview-control helper guard (U1). Draai via npm run test.
  • Handmatige end-to-end per modus, op twee surfaces (een lokaal .html-bestand én een draaiende localhost dev-app):
    • Modus 1: selecteer + comment → snippet + route in de chat.
    • Modus 2: highlight + inline edit → "verander X → Y" in de chat; pagina niet permanent gewijzigd.
    • Modus 3: teken + "voeg toe aan chat" → screenshot mét tekening in de chat via Alt+V.
  • Guard: zonder actieve claude-chat blokkeert verzending in elke modus en meldt de UI dat.
  • Security-regressie: bevestig dat er geen preload aan de guest is toegevoegd en hardenWebviewPreferences ongewijzigd is (test/webviewSecurity.test.ts blijft groen).
  • Grote payload: een grote outerHTML-snippet komt volledig in de chat (geen ~1 KB-truncatie).

Definition of Done

  • Alle drie de modi werken op zowel een lokaal .html-bestand als een draaiende dev-server.
  • Tekst (modus 1/2) landt via terminal.write in de actieve Claude-chat; grote snippets truncaten niet.
  • Beeld (modus 3) landt via clipboard.writeImage + Alt+V, met de route als begeleidende tekst.
  • De "geen actieve chat"-guard werkt in elke modus.
  • Geen preload/bridge in de guest; webview-hardening en test/webviewSecurity.test.ts intact; geen SDK/API.
  • Alle nieuwe unit tests groen; npm run test en de ESLint-gate slagen.
  • Hotkeys: bewust niet toegevoegd (uit scope).

Appendix — Risks & Sources

Risks & open items

Risk · mitigated

terminal.write ~1 KB-truncatie. Gemitigeerd door chunking (KTD5, U3). Verifieer met een expliciet grote-payload-test.

Risk · te verifiëren in implementatie

capturePage pakt de geïnjecteerde canvas mee (modus 3) en Alt+V programmatisch triggeren — beide best-effort; fallback = gebruiker drukt Alt+V. Bevestig het gedrag op de repo's Electron-versie vóór U6 afgerond wordt.

Deferred (uit origin)

console-message-kanaal voor auto-popover-op-selectie · batchen van annotaties · externe/SaaS-URLs · markdown/Milkdown-surface · annotatie-persistentie. Kosten nu van host-pull: één extra klik per annotatie.
