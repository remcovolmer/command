Triage-groepen

GESLOTEN Groep A — Niet-geauthenticeerd gast→host-kanaal opgelost · 284d150

Findings #1 + #2 deelden één oorzaak: onConsole (deps [], altijd actief) verwerkte elke console.log van de gast met een bekende constante sentinel — geen origin-, mode- of gesture-check. Elke pagina (inclusief externe URL's) kon de signalen vervalsen.

Toegepast Het kanaal is nu geauthenticeerd met een per-document nonce: de host genereert per document een random token (crypto.getRandomValues), injecteert het function-scoped als var ccNonce (nooit op window → onleesbaar voor pagina-scripts), en de gast echoot het in elk signaal. De parsers weigeren elk bericht met verkeerde/afwezige nonce (leeg = altijd weigeren, dekt het venster vóór dom-ready). Comment/markup zijn bovendien mode-gegate. De statische-scripts-invariant blijft intact (de nonce is host-randomness, geen page-data). Restrisico: console.log-hook + uitgelokte user-actie kan de nonce alsnog lekken — smaller, interactie-afhankelijk, gedocumenteerd in de code.

UITVOEREN Groep B — Robuustheid van de directe file-edit P2

Findings #5 (emoji, ✓ gefixt) en #6 (blinde DOM-splice) raken dezelfde edit-pijplijn. applyDomEdit splitst puur op indexPath+tag zonder inhoudscheck, dus bij live-DOM-drift (of een vervalst indexPath:[], tag:'html') overschrijft het de verkeerde/hele bron-range en de veiligere tekst-fallback draait niet. Verifieer consistentie vóór splice en weiger whole-document targets — maar een naïeve "match payload.before" is onveilig (innerText ≠ bron); gebruik een genormaliseerde tekstvergelijking. Hoort bij de #1-fix.

Findings

#1 · Vervalste save-sentinel overschrijft in-project bestanden P0 manual

WaarBrowserTab.tsx:244 → handleEditSave · reviewers: security, adversarial · confidence 100 · validator ✓
Wat breektEen pagina kan met console.log('__CC_ANNOTATE_SAVE__'+JSON…) — zonder klik — een edit forceren. Het schrijfpad nam payload.url (page-controlled) als doel; fs:writeFile is via validateFilePathInProject begrensd tot geregistreerde projecten, maar dat is projectbreed, niet het open bestand. applyDomEdit splitst bovendien blind op indexPath+tag (validator: indexPath:[]+tag:'html' overschrijft het hele bestand). Overschreven broncode draait bij de volgende build → code-execution/supply-chain-vector.
Responsgesloten Schrijf gebonden aan getURL() (d94e5a1) én kanaal geauthenticeerd met per-document nonce (284d150) — een pagina zonder de (onleesbare) nonce kan geen save-signaal meer forgen.

#2 · Vervalste comment/edit-sentinel injecteert een prompt in de actieve chat P0 manual

WaarBrowserTab.tsx:110 → handleCommentSend → terminal.write · security P0, adversarial P1 (→ P0 aangehouden) · confidence 100 · validatie degraded (over-bepaald)
Wat breektZelfde kanaal: een vervalste __CC_COMMENT__ zet zonder gesture attacker-tekst in de input van de actieve Claude-chat. terminal.write is niet bracketed-paste-gewrapt (net als de bestaande Ctrl+V-plak), dus een \r in het veld werd rauw doorgegeven en writeToTerminal behandelt \r/\n als submit → mogelijke auto-verzending naar een agent die commando's draait.
Responsgesloten Controlebytes (\r/ESC) uit page-velden gestript (d94e5a1, geen auto-submit/escape meer) én kanaal-nonce (284d150) — no-gesture-injectie is geblokkeerd; comment/markup ook mode-gegate.
Residual: console-message vuurt ook voor sub-frames/iframes — een third-party embed in een verder vertrouwde localhost-pagina kan de sentinels óók vervalsen. En clipboard:writeImage is via een vervalste __CC_MARKUP_ADD__ op dezelfde manier bereikbaar (ongevraagde screenshot naar klembord).
